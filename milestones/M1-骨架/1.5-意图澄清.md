# Step M1.5 意图澄清（模糊任务先问后做）

> 注：本步复用 M1.4 建立的「控制工具 → 解析 → 提前返回 `AgentResult`」模式。运行期上，澄清是 **前置闸门**，在 PLAN（M1.4）/执行之前；CLI（M1.6）负责收集答案回填后重跑。

> 实现说明（2026-07-17 初版；2026-07-17 同日重构）：控制工具的 schema **不再内联在 `loop.py`**，统一收口到新增的 `agent/core/control_tools.py`（`collect_control_tools(settings, plan_mode, has_plan)` 按模式取用，`update_plan` 仅「非 plan 模式 + 已知 plan_path」时并入）；系统提示外置为结构化文件 `agent/prompts/system.md`（frontmatter + Jinja2 模板，由 `agent/core/prompts.py` 加载）；循环改用 `model.stream()` 流式输出并实时落 `text` 事件；对话历史由**会话层持有**：`run(task, messages=None, *, clarify_total=0)` 接收本次会话历史、`AgentResult.messages`/`AgentResult.clarify_total` 回传，loop 无状态，支持多轮澄清回填。M1.4 直接复用 `control_tools.py` 与 `prompts/system.md`，无需重做基础设施。

## 实现方案

- **目标**：当任务**模糊 / 信息不足 / 存在多种合理走向**时，Agent 先通过控制工具 `ask_clarification` 向用户提出澄清问题（支持选项式多选题），**不执行任何动作**，提前返回 `AgentResult(needs_clarification=True, questions=...)`。上层（CLI）收集答案、拼成 user 消息后再次 `run`，模型据此继续。避免「猜错意图 → 盲改一通」。
- **改动文件**：
  - `agent/core/model.py`：协议 `Model.act/stream` 支持 `tools` 参数；`OpenAICompatibleModel` 非空才透传 `tools=`；`FakeModel`/`RecordingModel` 记 `tools_seen`。
  - `agent/core/control_tools.py`（新增）：**集中定义**所有控制工具 schema（`ASK_CLARIFICATION_TOOL`/`PRESENT_PLAN_TOOL`/`UPDATE_PLAN_TOOL`）与名常量，并导出 `collect_control_tools(settings, *, plan_mode, has_plan)` 按模式收集。`loop.py` 不再内联任何工具定义。
  - `agent/core/intent.py`（新增）：`Question` 数据结构、`extract_clarify(decision)` 解析器；仅从 `control_tools` 复用 `ASK_CLARIFICATION_TOOL_NAME`（不再持有工具 schema）。
  - `agent/core/prompts.py`（新增）+ `agent/prompts/system.md`（新增）：系统提示外置为 frontmatter + Jinja2 模板，`_system_prompt()` 委托 `load_prompt("system").render(...)` 渲染（注入 `clarify_enabled`/`plan_mode`/`has_plan`）。
  - `agent/core/loop.py`：`_decide` 改走 `model.stream()`（逐片 `text` 事件 + 收尾 `Decision`），模型调用前临时拼接 system 提示、不写入会话历史；`run(task, messages, *, clarify_total)` 接收会话历史、回传 `AgentResult.messages`/`AgentResult.clarify_total`，**loop 无状态**不持有对话；`_model_tools()` 委托 `collect_control_tools()`；澄清闸门逻辑不变。
  - `agent/core/events.py`：`Event.type` 增加 `"clarify"`；`Event` 增加 `questions`。`text` 事件（流式）复用了既有 `text` 字段。
  - `agent/config/settings.py`：新增 `clarify_enabled: bool = True`、`max_clarify_rounds: int = 2`、`clarify_hint_min_chars: int = 0`。
  - `tests/test_intent.py`：覆盖澄清提前返回/重跑/防呆/关闭/序列化；`test_loop.py::test_basic_flow_records_events` 增加流式 `text` 事件断言。
- **依赖/环境**：依赖 M1.3 的 `AgentLoop`/`EventStream`/`Decision`；新增运行依赖 `jinja2`（提示词模板）。

## 意图澄清机制（核心）

#### 1. 控制工具 `ask_clarification`（对齐 Claude Code 的 AskUserQuestion）
```json
{
  "type": "function",
  "function": {
    "name": "ask_clarification",
    "description": "当用户任务模糊、缺少关键信息或存在多种合理走向时，先向用户提问。不要在未澄清前执行任何操作。",
    "parameters": {
      "type": "object",
      "properties": {
        "questions": {
          "type": "array",
          "items": {
            "type": "object",
            "properties": {
              "question": {"type": "string"},
              "options": {"type": "array", "items": {"type": "string"}},
              "multiSelect": {"type": "boolean"}
            },
            "required": ["question"]
          }
        }
      },
      "required": ["questions"]
    }
  }
}
```

- `Question` 结构：`question: str`（必填）、`options: list[str] | None`（可选，给出候选项降低用户负担）、`multiSelect: bool = False`（多选开关）。
- 循环在 `clarify_enabled` 时把该工具并入 `tools`，并在 system 提示中告知：「若任务模糊或关键信息缺失，先调用 `ask_clarification` 澄清，不要猜测后直接动手。」

#### 2. 前置闸门 + 防呆上限
```python
# 循环顶部，每次 act 之后、plan/execute 之前最先判断
if self.settings.clarify_enabled and (cq := extract_clarify(decision)) is not None:
    self._clarify_rounds += 1
    stream.append(Event(type="clarify", ...))
    if self._clarify_rounds > self.settings.max_clarify_rounds:
        # 防呆：模型反复追问 → 不再 early-return，改为继续（或抛 ClarifyExhausted）
        # 默认策略：继续进入正常执行，把已有上下文交还给模型自行决策
        pass
    else:
        return AgentResult(needs_clarification=True, questions=cq, ...)
```

- **前置性**：澄清检查在 plan（`present_plan`）与执行**之前**，保证「先对齐意图，再计划/动手」。
- **防呆**：`max_clarify_rounds`（默认 2）封顶，避免模型陷入「反复问 → 用户答 → 又问」死循环；超出后让循环继续（也可配置为抛 `ClarifyExhausted` 由上层决定）。

#### 3. 答案回填（外层职责，M1.6 落地）
澄清返回后，外层把 `questions/answers` 拼成一条 user 消息重新 `run`：
```
用户已回答：
- Q: <question1>  A: <answer1>
- Q: <question2>  A: <answer2>
请基于以上澄清继续。
```
模型据此进入 plan（若 plan_mode）或直接执行。`clarify_rounds` 在重跑时**重置**（新一轮对话）。

#### 4. 轻量启发式（可选，非强制）
`clarify_hint_min_chars > 0` 时，若 `len(task) < clarify_hint_min_chars`，在 system 提示中**强化**澄清倾向（如「任务很短，请优先确认意图」）。最终是否澄清仍由模型决策——启发式只调权重，不替代决策（符合 98/1.6 法则）。

## 主流程算法（伪代码，含与 M1.4 的组合）

```text
async def run(self, task):
    messages = [system_prompt(self.plan_mode, self.settings.clarify_enabled),
                Message(role="user", content=task)]
    stream = EventStream()
    tools  = self._model_tools()
    self._clarify_rounds = 0

    for i in range(settings.max_iterations):
        decision = await model.act(messages, tools=tools)
        stream.append(Event(type="decision", decision=decision))

        # ① 澄清闸门（本步，最前）
        if self.settings.clarify_enabled and (cq := extract_clarify(decision)) is not None:
            self._clarify_rounds += 1
            stream.append(Event(type="clarify", ...))
            if self._clarify_rounds <= self.settings.max_clarify_rounds:
                return AgentResult(needs_clarification=True, questions=cq, ...)
            # 超出上限：放过，继续后续闸门

        # ② 计划闸门（M1.4）
        if self.plan_mode and (plan := extract_plan(decision)) is not None:
            stream.append(Event(type="plan", text=plan))
            return AgentResult(plan=plan, ...)

        # ③ 正常执行
        if decision.is_final:
            return AgentResult(text=decision.text or "", ...)
        results = await self._exec_tools(decision.tool_calls, stream)
        # ... 卡死检测 + 回填（同 M1.3）
    raise LoopMaxIteration(...)
```

### 模块边界
- `control_tools.py` 是**控制工具 schema 的单一事实来源**，`loop.py` 只调用 `collect_control_tools()` 取用，不再内联任何工具定义（回应「工具不要都放 loop」）。
- `intent.py` 是**纯解析 + 数据结构**，无 IO、不调用模型；`prompts.py` 是**纯加载器**；`loop.py` 只负责编排。
- 澄清问题**不下发执行**：`ask_clarification` 是控制工具（no-op），命中即返回，绝不进 `_exec_tools`。
- 答案回填与多轮交互在 CLI（M1.6），不在 loop——loop 保持「单次 run = 一轮澄清或一轮执行」的纯粹性；但**对话历史由会话层持有**：CLI 用「带上旧 `messages`、以答案为新 task 再次 `run`」续上上下文（回应「messages 历史」），loop 实例本身不保存任何对话（回应「messages 仅本次会话」）。

### 流程图（同计划，实现后确认无误）

**意图澄清主流程（前置闸门）**
```mermaid
flowchart TD
    A([run task]) --> B[注入 clarify 提示<br/>+ ask_clarification 控制工具]
    B --> C{model.stream}
    C --> D{Decision 含 ask_clarification?}
    D -- 是 --> E{clarify_total <= max?}
    E -- 是 --> F[emit clarify 事件<br/>early return]
    F --> G([AgentResult(needs_clarification=True, questions)])
    G -. 用户回答 .-> H[注入答案 user 消息, 重跑]
    E -- 否 --> I[放过, 进入后续闸门]
    D -- 否 --> I
    I --> J{计划闸门 / 执行}
```

### 实现原理
1. **前置闸门优于盲做**：模糊任务先对齐意图，再计划/执行，符合「配置与代码分离、AI 只决策」——模型决定「是否需要澄清」，确定性层负责拦截与回填。
2. **控制工具复用模式**：与 M1.4 `present_plan` 同构（控制工具 → 解析 → 提前返回 `AgentResult`），`intent.py` 只新增「问题结构 + 解析器」，循环编排零新增范式。
3. **防呆上限防死循环**：`max_clarify_rounds` 是确定性护栏，对应 M1.3 的 `max_iterations` 思想——AI 可能反复追问，确定性层兜底终止。
4. **loop 纯粹性**：单次 `run` 只处理「一轮澄清或一轮执行」，答案回填交 CLI；保证 `AgentLoop` 可单测、事件流可重放。

### 约定与决策理由
- 澄清问题**结构化但宽松**：`question` 必填、`options` 可选。给选项降低用户认知负担；不给也能工作（用户自由答）。
- 默认开启 `clarify_enabled`：通用 Agent 面向开放任务，模糊输入常见；CLI 可用 `--no-clarify` 关闭（M1.6）。
- 系统提示**外置为结构化文件** `agent/prompts/system.md`（YAML frontmatter + Jinja2 模板），由 `agent/core/prompts.py::load_prompt` 加载并按运行时变量渲染（`clarify_enabled`/`plan_mode`/`has_plan`）。与代码分离、可版本化、便于不同项目覆盖。
- 启发式 `clarify_hint_min_chars` 只调权重、不替代模型决策——避免「短任务一律被拦」的误杀（本步仅落配置位，`system.md` 未接此启发式，留 M1.6/后续按需强化）。

### 踩坑（实现后回填）
- **控制工具集中化**：初版把 `ASK_CLARIFICATION_TOOL` schema 内联在 `loop.py`，违反「工具不要都放 loop」。重构后所有控制工具 schema 与 `collect_control_tools()` 收口到 `agent/core/control_tools.py`，`loop.py` 零内联工具定义；M1.4 的 `present_plan`/`update_plan` 也在此定义，**`update_plan` 仅「执行期且已知 plan_path」（`has_plan and not plan_mode`）并入**，契合「update_plan 用于更新计划进度、应在非 plan 模式使用」的设计约束。
- **循环改用流式（`model.stream`）**：`_decide` 现在消费 `model.stream(messages, tools=...)`，逐片 yield `StreamEvent(type="text")` 并实时落 `Event(type="text", text=...)`，收尾 `done` 事件携带完整 `Decision`。因此 `test_basic_flow_records_events` 的事件序列多出 `text` 事件（在最终 `decision` 之前）。流式 token 事件对 CLI 实时渲染与 M5 可观测至关重要。
- **对话历史由会话层持有（loop 无状态）**：初版 `messages` 为 `AgentLoop` 实例字段、跨 `run()` 隐式累积，违背「messages 仅本次会话」。改为 `run(task, messages, *, clarify_total)` 接收会话历史、返回 `AgentResult.messages`（更新后的 user/assistant/tool 列表，不含 system）；system 提示由 loop 临时拼接、不写入会话历史。澄清回填即「会话层用答案作为新 task、带上旧 `messages` 再次 `run`」，跨 run 连续性完全由会话层决定（M5 的 sqlite 会话可直接持久此列表）。
- **跨 run 累计 vs 重跑重置**：计划原写「`clarify_rounds` 重跑时重置」，但这会让 `max_clarify_rounds` 防呆**永远不可达**（每次 run 首轮即 early-return，计数恒为 1）。改为**会话级累计 `clarify_total`**：会话层在多次 `run` 间传入/回传 `clarify_total`，前 `max` 次 early-return，第 `max+1` 次起不再 early-return（落入执行，`ask_clarification` 作为未知工具被降级，迫使模型 final 或触顶 `max_iterations）。测试 `test_anti_stuck_after_max_clarify_rounds` 据此编写（在三次 `run` 间 threading `messages`/`clarify_total`）。
- **`extract_clarify` 优先澄清、忽略同轮其它调用**：若模型一次 `Decision` 同时含 `ask_clarification` 与其它工具调用，直接返回 questions 让循环提前返回，同轮其它调用被丢弃（符合「先对齐意图再动手」）。`questions` 为空/缺失则视为无效，返回 `None` 让循环继续。
- **clarify 事件用 `questions: list[dict]` 而非 `list[Question]`**：保持 `events.py` 不反向依赖 `intent.py`，且 JSON 往返零额外序列化逻辑（dict 直存直取）。`AgentResult.questions` 仍用 `list[Question]`（运行时强类型）。
- **`clarify_enabled=False` 的健壮性**：此时 `collect_control_tools` 不并入 `ask_clarification`，但若模型「不听话」仍返回该 tool_call，循环不 early-return，而是走执行分支 → `UnknownTool` 降级为 `ToolResult(ok=False)`，不崩。测试 `test_clarify_disabled_does_not_early_return` 覆盖。
- **system 提示加载**：`_system_prompt()` 委托 `load_prompt("system").render(...)`；frontmatter 用 `yaml`、正文用 `jinja2.StrictUndefined`（未提供变量即报错，防止漏传 `plan_mode`/`has_plan`）。`agent/prompts/` 位于包内，随 `hatchling` 打包。
