# Step M1.6 CLI 入口 + 最简可观测

> 注：本步原编号为 **M1.4**，因在 M1.3 与（原）M1.4 之间插入了 **M1.4 PLAN 模式**、**M1.5 意图澄清**，故顺延为 **M1.6**。本步负责把前述能力（ReAct 循环、事件流、PLAN、意图澄清）接到 CLI，并在 run 结束时打印最简 trace 占位（完整可观测见 M5）。

## 实现方案
- **目标**：用 `typer` 实现 `run "<task>"` 与 `chat`（REPL）。`run` 支持 `--plan`（进入 plan 模式：先产出计划、等用户确认后再执行）与交互式澄清（模型 `ask_clarification` 时，CLI 收集答案回填后继续）。接 M1.1 的占位 `Tracer`，在 run 结束时打印最简 trace（span 树：agent.run → model.act → tool.exec，含耗时/父子）。
- **改动文件**：
  - `agent/cli.py`（`typer` 应用，`run`/`chat` 子命令；退出码 0/非0）：
    - `run`：`--plan/--no-plan` 控制 plan 模式（默认取 `settings.plan_mode`）；`--no-clarify` 关闭意图澄清；`--yes` 跳过计划确认直接执行。
    - 当 `AgentResult.needs_clarification`：交互/REPL 模式向用户逐题提问并收集答案，拼成 user 消息后再次 `loop.run`；非交互（`run` 且无 TTY）时直接报错退出（提示需交互或提供答案）。
    - 当 `AgentResult.plan` 且非 `--yes`：打印计划，等待确认（chat/交互）；非交互默认不执行（或 `--yes` 直接执行）。
  - 在 `loop.py` 中包裹 `tracer.span("agent.run", parent=None)`，工具执行包 `span("tool.exec", parent=agent_span)`。
  - `tests/test_cli.py`（用 `CliRunner` 或 subprocess + FakeModel，断言退出码与输出）。
- **关键接口**：
  ```python
  @app.command()
  def run(task: str, plan: bool = False, yes: bool = False, no_clarify: bool = False): ...
  ```

## 验收标准
- [ ] `python -m agent.cli run "写个 hello.txt"` 在 FakeModel 下跑通并退出码 0；输出含最终文本与最简 trace。
- [ ] `--plan` 下：FakeModel 先产出计划（`AgentResult.plan` 非空），CLI 打印计划并（交互/`--yes`）后进入执行；退出码 0。
- [ ] 模型 `ask_clarification` 时：交互模式能收集答案并继续；非交互（`run`，无 TTY）默认不静默跳过（报错退出）。
- [ ] `pytest tests/test_cli.py` 通过。
- [ ] trace 输出能体现父子关系（tool.exec 的 parent 是 agent.run）。

## 知识沉淀

- **新增模块**（`agent/cli.py`）：`typer` 应用，`run`/`chat` 子命令；`python -m agent.cli` 可运行（`if __name__=="__main__": app()`）。
- **接口签名**：
  - `run(task, plan: bool = Option(False,"--plan/--no-plan"), yes: bool = Option(False,"--yes"), no_clarify: bool = Option(False,"--no-clarify"))`；退出码 0/1/2，结束打印 `tracer.render()`。
  - `chat()`：REPL，单 `Session` 累积 `messages`，`exit/quit` 退出；任意轮次 `/plan` `/exec` `/approve` `/mode` 切换/查看模式。
  - `_build_model(settings)`：模块级工厂，构建生产 `OpenAICompatibleModel`；**测试 monkeypatch `agent.cli._build_model` 注入 FakeModel**（无测试专用分支）。
  - `Session`（会话状态持有者，位于 **`agent/core/session.py`**，不在 CLI）：持有 `messages`/`clarify_total`/`plan_mode`/`plan_path`/`loop`；`Session.step(task, ui, *, yes, fatal_plan_decline)` 处理澄清回填、计划确认与模式切换（确认后切 EXEC 续跑、未确认在 run 下退出 code 1、在 chat 下留在 PLAN 继续）。人机交互经 `SessionUI`（Protocol）注入，core 层不依赖 typer。
  - `SessionUI`（`agent/core/session.py` 的 Protocol）+ `_TyperUI`（`agent/cli.py` 的实现）：`interactive` 属性 + `ask`/`show_questions`/`show_plan`/`confirm_plan`/`notify`，把会话编排所需 IO 与 CLI 框架解耦（`_TyperUI` 承载原 `_print_plan` 与 `typer.prompt/confirm/echo`）。

> **注（架构已演进）**：本里程碑当时的 `SessionUI` + `_TyperUI` / `_RichPresenter`(`LoopPresenter`) 已被 **M-refactor 统一传输层** 合并为单一协议 `AgentTransport`（由 `TerminalTransport` 实现，见 `agent/core/transport.py` / `agent/runtime/terminal_transport.py`）。`Session.step` 现接收 `transport: AgentTransport`（不再有 `ui`/`presenter` 两个参数）。下文及 `knowledge/INDEX.md` 中若见 `SessionUI`/`LoopPresenter`/`_TyperUI`，均指该里程碑当时的命名，当前代码以 `AgentTransport`/`TerminalTransport` 为准。
  - `AgentLoop.run` 新增 `plan_mode`/`plan_path` 可覆盖入参（为 `None` 回落构造期缺省）：**plan/exec 模式按轮次传入，可在任意轮次自由切换**，loop 实例不保存模式状态。
  - `AgentLoop` 改动（`agent/core/loop.py`）：`run` 包裹 `tracer.span("agent.run")`；`_exec_tools` 每工具 `span("tool.exec", parent=agent_span)`；`_decide` `span("model.act", parent=agent_span)`；`agent.run`/`tool.exec` 父子树由 `Tracer.render()` 输出（M1.6 验收：父子关系）。
  - `AgentResult` 已含 `needs_clarification/questions`（M1.5）、`plan/plan_path/plan_steps/needs_plan_confirm`（M1.4）——CLI 直接消费。
- **渲染层（rich）**：`_RichPresenter`（`agent/cli.py`，实现 `agent/core/presenter.py` 的 `LoopPresenter` 协议）把 ReAct 循环内部事件实时渲染成交互终端输出，**区分「思考/输出/工具调用」**：`reasoning`（思考）暗色实时打印、`content`（输出）用 `rich.live.Live` 逐片渲染 Markdown、工具调用/结果用 `Panel`。`Session.step(task, ui, *, presenter=...)` 透传 presenter 到 `AgentLoop.run(presenter=...)`；loop 在 `_decide`（流式文本）与 `_exec_tools`（工具调用/结果）回调 `on_text`/`on_tool_call`/`on_tool_result`；core 层不依赖 rich。非终端（CI / CliRunner）降级：`_use_live = console.is_terminal` 为 `False` 时仅累积文本、`close()` 一次性渲染 Markdown，使测试可断言最终文本（且 `run`/`chat` 既有的「最终文本出现在 stdout」断言仍可过）。
- **token 用量（usage）**：`Decision.usage`（`prompt_tokens`/`completion_tokens`/`total_tokens`）由 `OpenAICompatibleModel` 从响应 `usage` 解析（流式需 `stream_options={"include_usage": True}`，取末个 usage-only chunk；`act` 用 `getattr(resp,"usage",None)` 容错假 client），并从 `delta.reasoning_content`（DeepSeek 思考）解析出 `kind="reasoning"` 的流式文本；`AgentLoop.run` 逐轮累加进 `AgentResult.usage`，**每轮 ReAct 循环结束**（`run`/`chat` 的 `step` 后）由 `_RichPresenter.report_usage` 打印。
- **模块边界**：`Session` 位于 **core**（`agent/core/session.py`），不依赖任何 CLI 框架；CLI 只做「配置→模型→构造 Session/UI→输出」的装配，并提供 `SessionUI` 的 typer 实现（`_TyperUI`）。对话状态由 `Session` 在多次 `step` 间持有并传入/回传（呼应「messages 仅本次会话」）；澄清/计划/执行的串联在 `Session.step`，loop 保持无状态引擎（消息与模式均按 run 传入）。人机交互经 `SessionUI` 注入，使 `Session` 可脱离终端独立测试。
- **约定与决策理由**：
  - 澄清回填：交互模式 `typer.prompt` 逐题收集、拼成 user 消息再次 `loop.run`；**非交互（run 且无 TTY）默认报错退出（code 2），不静默跳过**（M1.6 验收）。
  - **plan/exec 任意轮次自由切换（用户诉求）**：模式是会话级、按轮次可变状态，由 `Session` 持有并通过 `AgentLoop.run(plan_mode=, plan_path=)` 传入；`chat` 中用 `/plan` `/exec` `/approve` 在任意轮次切换，无需重建会话。`run --plan` 仅作为**初始模式**，确认后自动切 EXEC 续跑同一任务。
  - plan 两阶段（run 默认路径）：plan 模式 loop 落盘 → CLI 打印 + 确认（`--yes` 或交互 confirm）→ exec 模式 loop（带 `plan_path`，启用 `update_plan`）。
  - trace 始终打印（即便失败），便于排障；完整可观测留 M5（OTel/Langfuse）。
  - 异常在 `run` 内 try/except 转退出码 1，避免未捕获栈。
- **踩坑（实现后回填）**：
  - 澄清问题打印到 stderr（`typer.echo(..., err=True)`），测试断言用 `result.output`（CliRunner 合并 stdout/stderr），不能断言 `result.stdout`。
  - `plan_mode` 缺省取 `settings.plan_mode`（`AgentLoop.__init__` 回落），CLI 显式传 `--plan` 覆盖——漏传会导致 plan 闸门不触发。
  - 提前返回（澄清/计划）前 loop 已 `conv.append(assistant消息)`，CLI 续接时历史连贯；final 也补 assistant。
  - `_build_model` 抽成模块级函数，测试注入 FakeModel 时不污染生产代码路径。
- **测试**：`tests/test_cli.py`（CliRunner + 注入 FakeModel）覆盖 run 跑通+退出码0+含 trace / `--plan`+`--yes` 两阶段 / 澄清非交互退出2 / trace 父子关系（`tracer.spans` 断言 `tool.exec.parent_id == agent.run.id`）；`tests/test_plan.py` 覆盖 M1.4 验收（落盘/进度回写/风险门控/非 plan 模式/事件往返）。全量 `pytest` 64 passed（含新增 usage 累加与 presenter 流式/工具回调测试）。
