# Step M4.2 Microcompact（零成本）

## 实现方案

**目标**：实现 Microcompact 压缩器——在每次 API 请求前，把 `conv` 中（`boundary = len(conv)`，即整个当前 conv）较旧的 `role="tool"` 消息的 `content` 替换为占位符 `[Old tool result content cleared]`，保留最近 N 个结果不动。零 API 调用，消息条数不变，不拆散 `tool_use`/`tool_result` 配对。独立作用于 auto-compact 的 `compact_boundary` 之外（详见下方关键决策）。

**改动文件**：
- `agent/context/compactors/microcompact.py`（新建）
- `agent/context/manager.py`（ContextManager 集成 Microcompact 调用入口）
- `tests/test_context.py`（追加测试用例）

**关键接口**：

```python
# agent/context/compactors/microcompact.py

COMPACTABLE_TOOLS = {"bash", "read", "grep", "glob", "write", "edit", "find"}
"""可压缩的工具结果类型（大输出类工具）。"""

PLACEHOLDER = "[Old tool result content cleared]"
"""占位符字符串，对齐 Claude Code 行为。"""


class Microcompact:
    """Microcompact 压缩器：零成本，仅替换旧 tool_result 内容为占位符。

    策略：
    - 只操作 compact_boundary 之前的消息（已被压缩过的不再处理）。
    - 只操作 conv 中 role="tool" 的消息。
    - 保留最近 keep_recent 个 tool_result 不动。
    - 不修改 tool_use 消息（配对完整性靠 tool_call_id 保持）。
    - 消息条数不变，索引不变。
    """

    def __init__(self, keep_recent: int = 5, compactable_tools: list[str] | None = None):
        self.keep_recent = keep_recent
        self.compactable_tools = compactable_tools or list(COMPACTABLE_TOOLS)

    def compact(self, conv: list[Message], boundary: int) -> list[Message]:
        """对 conv 执行 Microcompact，返回新列表（原地修改或新列表）。"""
        if boundary <= 0:
            return conv

        # 统计 boundary 前所有 tool 消息中属于可压缩工具的索引
        tool_indices: list[int] = []
        for i, msg in enumerate(conv[:boundary]):
            if msg.role == "tool" and self._is_compactable(msg):
                tool_indices.append(i)

        # 保留最近 keep_recent 个
        replace_indices = (
            set(tool_indices[: -self.keep_recent])
            if len(tool_indices) > self.keep_recent
            else set()
        )

        for i in replace_indices:
            conv[i] = Message(
                role="tool",
                content=PLACEHOLDER,
                tool_call_id=conv[i].tool_call_id,
            )
        return conv

    def _is_compactable(self, msg: Message) -> bool:
        """判断此 tool 消息的内容是否来自可压缩工具（通过 tool_call_id 交叉验证不可行，
        此处简化：只要有 content 且不太短的 tool 消息均视为可压缩。
        更精确的做法需遍历 assistant 消息交叉 tool_call_id→name，可选增强。"""
        return bool(msg.content) and len(msg.content) > 100
```

**集成到 ContextManager**：

```python
# agent/context/manager.py 新增
from agent.context.compactors.microcompact import Microcompact

class ContextManager:
    def __init__(self, ...):
        ...
        self.microcompact = Microcompact(
            keep_recent=context_config.microcompact_keep_recent,
        )

    def apply_microcompact(self) -> list[Message]:
        """执行 Microcompact，返回处理后的 conv。"""
        if not self.conv:
            return self.conv
        return self.microcompact.compact(self.conv, self.compact_boundary)
```

**关键决策**：
- **只改 content，不动配对**：`tool_call_id` 保持不变，API 协议满足。
- **保留最近 N 个**：默认 5（`keep_recent=5`），Claude Code 也是类似思路——最近的操作结果不应被压缩，否则模型看不到当前正在改的代码。
- **边界限定（已偏离草稿）**：Microcompact 作用于**整个当前 conv**（`boundary = len(conv)`），而非仅 `compact_boundary` 之前。原因：边界之前的内容已被 auto-compact 压成摘要（无 `role="tool"` 长结果可清），且会话前期 `compact_boundary=0` 会使 Microcompact 整体失效；它独立于 auto-compact 边界处理全量，活跃区的旧工具结果照常被清。
- **零成本**：纯字符串替换，不调 LLM。

**依赖/环境**：
- `Message` 数据类，`role="tool"` 和 `tool_call_id` 字段。
- `ContextManager.compact_boundary`（M4.1 实现）。

## 验收标准

- [ ] `Microcompact.compact(conv, 0)` 不修改任何消息。
- [ ] `Microcompact.compact(conv, len(conv))` 把 boundary 前旧 tool 消息替换为占位符，保留最近 5 个。
- [ ] `keep_recent=0` 时替换所有旧 tool 消息。
- [ ] 占位后 `tool_call_id` 不变，`role` 仍为 `"tool"`。
- [ ] `Message` 内容长度 > 100 才被替换，短消息不动。
- [ ] 非 `role="tool"` 的消息不被修改。
- [ ] `ContextManager.apply_microcompact()` 返回正确结果。
- [ ] 测试：`tests/test_context.py` 追加 ≥8 用例覆盖边界 / keep_recent / 占位符 / 非 tool 不动。

### 接口与常量（落到 `agent/context/compactors/microcompact.py`）

- `class Microcompact`：
  - `__init__(keep_recent: int = 5, compactable_tools: list[str] | None = None)`；`compactable_tools` 默认 `list(COMPACTABLE_TOOLS)`。
  - `async compact(conv, boundary) -> list[Message]`：原地替换 `boundary` 之前的旧 tool 内容为占位符，返回同一 `conv` 列表（**就地修改 + 返回同一引用**）。
  - `_build_id_to_name(conv) -> dict`：遍历 assistant 消息的 `tool_calls`，构建 `tool_call_id → 工具名` 映射（交叉验证用）。
  - `_is_compactable(msg, id_to_name) -> bool`：判断某 tool 消息是否应被占位。
- `COMPACTABLE_TOOLS = ("bash","read","grep","glob","write","edit","find")`（大输出类工具）。
- `PLACEHOLDER = "[Old tool result content cleared]"`（对齐 Claude Code）。

### 关键决策 / 踩坑（重要，M4.3 务必先看）

1. **协议对齐：`compact` 改为 `async`**（偏离 4.2 草稿的同步签名）。原因：`Compactor` 协议（`agent/context/compactors/__init__.py`）声明 `async def compact(...)`，且 M4.3 的 Auto Compact 必须 async 调 LLM 摘要。若 Microcompact 用同步签名，调用处 `await compactor.compact(...)`（统一入口）会报错。因此 **所有 Compactor 实现统一 async**，M4.1 测试里 `isinstance(Dummy, Compactor)` 检查通过。`ContextManager.apply_microcompact()` 相应为 `async`。
2. **`apply_microcompact()` 用 `boundary = len(self.conv)` 而非 `self.compact_boundary`**（偏离草稿）。草稿写的是传 `self.compact_boundary`，但会话初期 `compact_boundary=0` 会使 Microcompact 对整个 conv 无操作——违背「每次 API 请求前清除旧工具结果」的本意。Microcompact 独立于 auto-compact 的边界：auto-compacted 区域通常是摘要（非 role="tool" 长结果），不会被误伤。验收用例正是用 `compact(conv, len(conv))` 覆盖全量，故此处一致。
3. **`keep_recent` 的边界修正**（修复草稿 bug）：草稿 `tool_indices[:-self.keep_recent]` 在 `keep_recent=0` 时变成 `tool_indices[:0]`=空，反而一个都不替换，违背验收「keep_recent=0 替换所有」。改为 `tool_indices[: len(tool_indices) - self.keep_recent]`（仅当 `len(tool_indices) > keep_recent` 时替换）。
4. **精确工具名过滤（增强草稿简化版）**：草稿 `_is_compactable` 只用 `content>100` 判断。落地版增加交叉验证：通过 `tool_call_id → assistant.tool_calls[].name` 取到工具名，仅当在 `compactable_tools` 内才压缩；非大输出类工具（如 `AskUserQuestion` 回执）即便很长也不压缩。`compactable_tools` 为空时回退到「任意长 tool 结果都可压缩」，保持向后兼容。
5. **只改 `content`，保留 `tool_call_id` 与 `role="tool"`**，绝不拆散 `tool_use`/`tool_result` 配对（协议铁律）。消息条数、索引不变 → 不影响后续 auto-compact 的配对逻辑。
6. **零成本**：纯字符串替换，不调 LLM；`Microcompact` 零依赖（仅 `from agent.core.model import Message`），`ContextManager` 导入它无循环依赖。
