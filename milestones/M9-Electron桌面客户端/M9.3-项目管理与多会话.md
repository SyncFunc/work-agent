# Step M9.3 项目管理与多会话

## 实现方案

- **目标**：实现「项目根切换器 + 多会话标签页 + 会话列表 + resume/fork + 重放」，对齐 M7.4 的 `session.list`/`attach`/`switch`/`replay_start|end` 与 M7.5 的 `/resume` `/fork` 命令。

- **新增文件**（`desktop/src/`）：
  - `features/projects/ProjectSwitcher.tsx`：左侧项目根切换（输入/浏览选择目录；当前 `project_root` 持久化到 `localStorage`）。
  - `features/sessions/SessionTabs.tsx`：多会话标签页（每个 tab = 一个 `session_id` + `project_root`）。
  - `features/sessions/SessionList.tsx`：当前项目下会话列表（`session.list` → `session_list`）。
  - `features/sessions/useSessions.ts`：会话状态机（连接 `DaemonClient`，管理 `new/attach/switch/detach`、`replay` 缓冲）。
  - `features/sessions/ReplayBuffer.ts`：消费 `replay_start`→批量 `event`→`replay_end`，重建历史渲染（瞬时 `tool_call_delta` 不入 replay，与 daemon 一致）。

- **关键设计**：
  1. **项目切换**：切换 `project_root` → 调 `listSessions(projectRoot)` 刷新列表；当前激活会话随之切换（同一 daemon，不重启）。
  2. **多标签页**：每个 tab 记录 `{session_id, project_root}`；新会话 `newSession(name, projectRoot)` 开 tab 并 `attach`；
     切换 tab = `switch(session_id, projectRoot)`（触发 replay 回填历史）。
  3. **resume / fork**：
     - resume：`session.list` 选中历史会话 → `attach(session_id, projectRoot)`。
     - fork：调 `command('fork', session_id)`（M7.5 透传 `dispatch_command`）；若 M9.0 已实现 `command_result` 回传新 `session_id` 则直接开新 tab，否则 fork 后重新 `listSessions` 取最新并 `attach`。
  4. **重放**：`attach/switch` 后按 `replay_start/end` 区间把历史 `Event` 灌入渲染（M9.4 消费），保持瞬时事件排除规则。
  5. **空状态**：当前项目无会话时展示「新建会话」入口。

- **依赖/环境**：M9.0 daemon（携带 `project_root`）、M9.2 `DaemonClient`。

## 知识沉淀

### 会话状态机字段（`features/sessions/sessionMachine.ts`）
- `SessionsState = { tabs: SessionTab[]; activeId: string|null; replaying: boolean; byId: Record<string, AgentEvent[]> }`。
- `SessionTab = { id; name?; project_root; attached }`；`id` 对齐 daemon `session.created` 的 `session_id`（**不是** `session_id` 键，daemon `list_info` 用 `id`）。
- `upsertTab` 必须**保留已有 tab 的 name**：`attached`/`list` 消息可能不带 `name`，若用 `id.slice(0,8)` 覆盖会丢失用户命名。
- `liveEvent` 守卫：`state.replaying || activeId == null` 时**不**追加事件，避免 replay 期间瞬时事件重复渲染（与 daemon `_replay` 仅补非 transient 事件一致）。

### replay 缓冲合并策略（`features/sessions/replay.ts`）
- `ReplayBuffer.start/push/end` + `isActive`；`isReplayStart/isReplayEnd` 判断 `MsgType`。
- 客户端只认 daemon 重放的**持久化**事件（`tool_call_delta` 等瞬时事件 daemon 侧已排除），故无需客户端二次过滤。

### fork 最终回传方案
- M9.0 `dispatch_command` 未实现 `command_result` 回传新 `session_id`，故 `useSessions.forkSession` 走：`command('fork', id)` → `listSessions(projectRoot)` 取最新 → `attach(最新 id)`。后续若 daemon 补 `command_result` 可直接开 tab，无需轮询。

### daemon 就绪竞态修复（M9.0/M9.1 联动）
- `server.py` 原本在 `asyncio.run(_serve(...))` **之前**打印 `ws=...` 就绪日志，导致 `DaemonManager.waitForReady`（解析日志）在端口尚未可连接时误判就绪。
- 已把就绪日志移入 `_serve` 的 `async with create_ws_server(...)` 块**内**，确保日志只在服务真正监听后出现。

### 真机冒烟脚本要点（`scripts/smoke-sessions.mjs`）
- 需 Node 22+ 全局 `WebSocket`；`session.new` 请求必须在 `ws.onopen` 内发出（否则 `Sent before connected`）。
- daemon 按 `project_root` 构建 Model，缺 `llm.api_key` 会在 `session.new` 报 `handler_error`；冒烟为每个临时项目写入 `.agent/settings.yaml` 占位 key（不真正调用模型）。
- 多项目隔离由 `registry.new(project_root, ...)` + `list_info(project_root)` 保证；本脚本验证 A=2/B=1。
- 仅本地验证（非 CI 必跑）。
