# Step M9.2 TS 协议客户端库

## 实现方案

- **目标**：在 `desktop/src/protocol/` 实现与 `agent/daemon/protocol.py` 对齐的 TypeScript 消息类型与 WS 封装，
  作为所有功能面板（M9.3–M9.7）与 daemon 通信的唯一入口。并加一份**契约测试**防止 Python/TS 漂移。

- **新增文件**（`desktop/src/protocol/`）：
  - `types.ts`：`MsgType` 字面量联合类型、`Envelope` 信封类型、`Event` 反序列化类型（对齐 `agent/core/events.py` 的 `to_dict` 字段）。
  - `client.ts`：`DaemonClient` 类——连接/重连/发送/订阅；把 `event` 消息解析为 `Event`，HITL 请求做 `id` 配对回传。
  - `events.ts`：`Event.from_dict` 的 TS 等价：`parseEvent(dict): Event`（含 `decision/tool_use/tool_result/questions/plan_update` 嵌套结构）。
  - `contract.test.ts` + 配套 `desktop/contract/msgtype.json`（或从 Python 生成的快照）：断言 TS `MsgType` = Python `MsgType`。

- **关键设计**：
  1. **`MsgType` 联合**（与 `protocol.py` 完全一致，含 C→S 与 S→C 全部值）：
     `hello | session.new | session.attach | session.switch | session.detach | session.list | task.send | answer | confirm_plan | approve | command | welcome | session.created | attached | detached | session_list | event | replay_start | replay_end | ask | show_questions | show_plan | show_skills | show_agents | notify | usage | close | error`。
  2. **信封**：`{ type: MsgType, id?, session?, payload: Record<string, unknown> }`，`JSON.stringify` 收发。
  3. **`DaemonClient` API**（供面板调用）：
     - `connect()` / `onClose(cb)` / 自动重连（指数退避，上限 ~5s）。
     - `hello(token)`、`newSession(name, projectRoot)`、`attach(sessionId, projectRoot)`、`switch(sessionId, projectRoot)`、`listSessions(projectRoot)`、`detach()`。
     - `sendTask(text, {yes?, plan?, projectRoot?})`、`answer(id, text)`、`confirmPlan(id, confirmed)`、`approve(id, approved)`、`command(name, args)`。
     - 事件订阅：`onEvent(cb: (ev: Event) => void)`、`onMessage(type, cb)`（HITL：`ask`/`show_plan`/`show_skills`/`show_agents`/`notify`/`usage`/`close`/`session_list` 等）。
  4. **Event 反序列化**：`events.ts` 按 `agent/core/events.py` 字段重建：保留 `seq/type/ts`，
     嵌套 `decision{text,tool_calls[{id,name,arguments}]}`、`tool_use`、`tool_result{ok,output,error}`、
     `questions[]`、`plan_update{step_id,status,note}`、`text`/`kind`/`tc_*` 等。**这是 M9.4 渲染的数据源**。
  5. **契约测试（防漂移）**：
     - 在 `agent/tests/` 或 `desktop/` 放一个 pytest/脚本：读取 `desktop/src/protocol/types.ts` 中的 `MsgType` 字面量，
       与 `import agent.daemon.protocol.MsgType` 的取值集合比对，不一致则失败。
     - 简化实现：在 `agent/tests/unit/test_m9_protocol_contract.py` 里 `subprocess` 调 `node -e` 解析 TS 文件，
       或维护 `desktop/src/protocol/msgtype.generated.json` 由 Python 端 `MsgType` 生成、TS 引用之。

- **依赖/环境**：`ws`（或 Electron 内置 `WebSocket`）、`typescript`。测试用 `vitest`/`jest`。

## 知识沉淀

> 来源：M9.2 TS 协议客户端库。

- **MsgType 联合（单一事实来源）**：`desktop/src/protocol/types.ts` 用 `export const ALL_MSG_TYPES = [...] as const` 作为运行时集合，`export type MsgType = (typeof ALL_MSG_TYPES)[number]` 由它派生——改协议只动 `ALL_MSG_TYPES` 一处。与 `agent/daemon/protocol.py` 的 `MsgType(StrEnum)` 28 个值逐一对应（C→S 11 + S→C 17）。`Envelope = { type, id?, session?, payload }`，与 `make_message`/`parse_message` 一致。
- **Event 反序列化字段映射（TS `events.ts` ↔ Python `events.py`）**：`parseEvent(dict)` 严格对齐 `Event.to_dict`/`from_dict`；`eventToDict(ev)` 反向（仅含非 null 字段）用于双向 fixture。`AgentEvent` 字段：`seq/type/ts`（`type` 为 `EventTypeStr` 11 值）、`transient?`、`decision?{text?,tool_calls:[{id,name,arguments}]}`、`tool_use?{id,name,arguments}`、`tool_result?{ok,output?,error?}`、`tool_call_id?`、`tc_index?/tc_name?/tc_args?`（tool_call_delta）、`text?/kind?`（text：reasoning|content）、`error?`、`questions?[{question,options?,multiSelect?}]`（对齐 `intent.Question.to_dict`）、`plan_path?`、`plan_update?{step_id,status,note?}`（对齐 `plan_progress` 事件）。嵌套解析对缺失/非法子结构做容错（返回 null 而非抛错，主事件仍解析）。
- **DaemonClient API**：`connect()`（open 即 `hello`，返回 Promise）、`close()`（manual 不重连）、`onMessage(type,cb)`（含 HITL：`ask`/`show_plan`/`show_questions`/`show_skills`/`show_agents`/`notify`/`usage`/`close`/`session_list` 等）、`onEvent(cb)`（收到 `event` 消息后 `parseEvent` 派发 `AgentEvent`）、`onClose(cb)`；协议方法 `hello/newSession/attach/switch/listSessions/detach/sendTask/answer/confirmPlan/approve/command`。`event` 消息的 `payload.event` 即 `Event.to_dict`；HITL 请求 `ask`/`confirm_plan`/`approve` 的 `id` 由客户端原样回传（`answer(id,text)` 等带 `id`）。`session.created`/`attached` 自动跟踪 `currentSessionId`+`project_root` 供重连恢复。
- **重连（指数退避）**：`onclose` 且非 `manualClose` → `scheduleReconnect()`：`delay = min(500*2**attempt, reconnectMax=5000)`；重连 `connect()` 后重新 `hello`，若 `currentSessionId` 存在则 `attach` 恢复订阅。注入 `WebSocketImpl` 便于测试（默认全局 `WebSocket`，浏览器/Node22 可用）。
- **契约测试机制（解析 vs 生成）**：选「解析源文件」方案——`check-msgtype.mjs` 直接正则抽取 Python `class MsgType` 块（到 `@runtime_checkable` 止）的 `NAME = "value"` 与 TS `ALL_MSG_TYPES = [...]` 字面量，比对集合；无中间生成文件，真正的单一事实来源即两源文件，漂移即失败。`tsconfig` 仅 `include: src/**`，所以 `*.test.ts` 也受 `tsc --noEmit`（即 `lint`）检查，需 `vitest` 类型（`import { describe,it,expect } from 'vitest'` 显式导入，不污染全局 `types`）。
- **环境**：`desktop` 引入 `vitest`（devDep）+ `vitest.config.ts`（`environment: 'node'`，`include: src/**/*.test.ts`）；`package.json` 加 `test`/`test:watch`。真机握手用 Node22 全局 `WebSocket`（`smoke-daemon.mjs`），无需额外 `ws` 依赖。
- **测试可达性（headless）**：GUI/重连真机在 headless 用注入假 `WebSocket` + `vi.useFakeTimers` 覆盖；真实 daemon 握手由 `smoke-daemon.mjs` 单独验证（已收到 welcome）。
