# Step M9.0 daemon 多项目感知与协议扩展（前置）

## 实现方案

- **目标**：当前 `start_daemon(settings)` 全局只 `load_settings()` 一次（项目根隐式为 cwd/`AGENT_PROJECT_ROOT`），
  `SessionStore` 路径固定（`settings.obs.sessions_db_path`，项目相对路径）。要支持「全局单一 daemon + UI 内切换项目根」，
  必须把 daemon 改为**按 `project_root` 隔离**：同一进程服务多个项目，各项目的 settings、会话 sqlite 互不串扰。

- **改动文件**：
  - `agent/daemon/registry.py`（`SessionRegistry` / `SessionHandle`）
  - `agent/daemon/server.py`（`_route` / `_attach` / `_switch` / `_command` / `_task_send` / `_replay` / `start_daemon` / 工厂）
  - `agent/daemon/protocol.py`（`MsgType` 无需新增类型，但 `session.new` / `session.attach` / `session.switch` / `session.list` / `task.send` 的 **payload 增加 `project_root`**；文档补充方向说明）
  - `agent/daemon/client.py`（CLI 客户端同步传 `project_root`，默认 cwd，保持向后兼容）
  - `agent/cli.py`（`daemon` 子命令：网络配置走用户级/内置，不再绑定项目根）

- **关键设计**：
  1. **双 settings 模型**：
     - daemon 自身网络配置（host/port/health_port/token）来自 `load_settings()`（不传 `project_root`，取用户级 `~/.agent/settings.yaml` 或内置默认）——与项目无关。
     - 每个会话运行时用 `load_settings(project_root=pr)` 解析项目级配置（含 `obs.sessions_db_path`、llm、sandbox 等）。
  2. **registry 增加 `project_root` 维度**：
     - 内部以 `(project_root, session_id)` 组织 handles；提供 `new(project_root, name)` / `attach(project_root, session_id)` / `switch(conn, project_root, session_id)` / `list_info(project_root)` / `get(project_root, session_id)`。
     - 维护 `project_root -> SessionStore` 缓存，避免重复打开 sqlite。
  3. **工厂签名变更**：`session_factory(project_root, session_id)`、`restore_factory(project_root, session_id)`，内部 `load_settings(project_root=...)` + `SessionStore(settings.obs.sessions_db_path)`。
  4. **协议 payload 扩展**（向后兼容：缺省 `project_root` 时回退 cwd，保持现有 CLI 行为）：
     - `session.new`: `{name?, project_root}`（必带；CLI 默认 cwd）
     - `session.attach`: `{session_id, project_root}`
     - `session.switch`: `{session_id, project_root}`
     - `session.list`: `{project_root}`（列出该项目下会话；返回 `sessions` 列表含 `session_id/name/updated_at`）
     - `task.send`: `{text, yes?, plan?, project_root?}`（会话已绑定 project_root，通常无需再带；保留以便未来单会话跨项目调用）
  5. **`/resume` `/fork` 命令**：现有 `session_command.py` 已支持；daemon 侧 `dispatch_command` 透传，前端经 `command` 消息调用即可（M9.3 接入）。fork 产出的新 `session_id` 由 `Session` 返回，需经协议回传（见下方「待确认扩展」）。
  6. **attach/switch 恢复**：某 `project_root` 下会话若不在内存，仍走 `restore_factory` 冷启动（M7.4 已落地的 replay 机制不变，仅增加 project_root 路由）。

- **依赖/环境**：M7 已落地 daemon 与 WS 协议；本步仅做维度扩展 + 测试。

- **待确认扩展（fork 回传）**：当前 `dispatch_command` 处理 `/fork` 后仅 `fb(...)` 文本反馈，未把新 `session_id` 回传前端。
  M9.0 建议在 `command` 响应里增加 `command_result`（含 `{ok, session_id?, message}`），使前端 fork 后能直接跳转新会话。
  若本期不做，M9.3 可用「fork 后重新 `session.list` 取最新」兜底。

## 知识沉淀

- **registry 双维度接口签名**：`new(project_root, name)` / `attach(conn, project_root, sid)` / `switch(conn, project_root, sid)` / `list_info(project_root=None)`；工厂 `session_factory(project_root, sid)`、`restore_factory(project_root, sid) -> Session | None`、`store_factory(project_root) -> SessionStore`。`SessionHandle.project_root` 字段。
- **双 settings 解析约定**：`start_daemon(settings)` 的 `settings` 仅取 daemon 网络项；每会话运行时 `load_settings(project_root=...)` + `store_for(pr)` 惰性缓存 `SessionStore`。相对 db 路径经 `_anchor_path(p, project_root)` 锚定到项目根（解决多项目串库）。
- **`project_root` 协议传递位置**：`session.new`/`attach`/`switch`/`list` 的 payload；`session.created`/`attached` 响应回带；`session.list` 响应 `{project_root, sessions}`。缺省回退 cwd 仅 CLI 兼容。
- **fork 回传方案取舍**：M9.0 未实现 `command_result`（标记待 M9.3）。`dispatch_command` 当前不含 `/fork`/`/resume`；M9.3 接 `/fork` 时可用「fork 后重新 `session.list`」兜底。
