# Step M9.5 HITL 模态与审批可视化

## 实现方案

- **目标**：把 daemon 的 HITL 协议消息渲染为模态交互，并可视化审批/沙箱决策，等价于 M8 HITL 模态 + 审批可视化。

- **协议消息（S→C，来自 M7.2/M7.3）**：
  - `ask{id, question}`（`agent/core/intent.Question` 结构：`questions[]` 选项或自由文本）→ 回传 `answer{id, text}`。
  - `show_plan{plan, plan_path, plan_steps[]}` / `confirm_plan` 实际是 `show_plan` 后由前端触发 `confirm_plan{id, confirmed}`（注意：M7 协议里 `CONFIRM_PLAN` 既作请求也作回传；daemon 在 `show_plan` 时等待前端 `confirm_plan`）。
  - `approve{id, action:{tool, risk, args, description, approval_request}}` → 回传 `approve{id, approved}`。
  - `show_questions` / `show_skills` / `show_agents` 为信息展示（非阻塞）。

- **新增文件**（`desktop/src/features/hitl/`）：
  - `HitlModalHost.tsx`：全局模态宿主，按 `id` 排队 HITL 请求（同 `id` 闭环）。
  - `AskModal.tsx`：多选/单选（`questions[].options`）或自由输入 → `answer`。
  - `PlanModal.tsx`：展示 `plan` 文件内容 + `plan_steps` 进度；「批准并继续 / 修改 / 拒绝」→ `confirm_plan{confirmed}`。
  - `ApproveModal.tsx`：展示 `action`（工具名/风险档/参数摘要）；「批准 / 拒绝」→ `approve{approved}`。
  - `ApprovalBadge.tsx` / `SandboxViz.tsx`：在工具块（M9.4）旁展示风险档（`tool` 事件的 `risk`/审批状态）与沙箱档位（`settings.sandbox.profile`）。

- **关键设计**：
  1. **id 配对**：`DaemonClient` 收到 `ask`/`show_plan`/`approve` 时按 `id` 弹模态；用户操作回传**同 `id`** 的 `answer`/`confirm_plan`/`approve`，daemon 侧 `BridgeTransport.resolve(id, value)` 唤醒 `Session.step`（M7 已落地）。
  2. **阻塞性**：HITL 模态出现期间，输入区禁用；模态关闭（回传）后才恢复。
  3. **审批可视化**：工具块标题显示风险档（如 `danger`/`elevated`/`safe` 配色）；沙箱档位在设置或状态栏展示（M9.6/M9.7）。
  4. **plan 进度**：`PLAN_PROGRESS` 事件（`plan_update{step_id,status,note}`）实时更新 `PlanModal` 的步骤状态，无需重开模态。

- **依赖/环境**：M9.2 `DaemonClient`（含 HITL 消息订阅）、M9.4 工具块（审批徽章挂载点）。

## 知识沉淀

### HITL id 配对最终形态（`features/hitl/`）
- `DaemonClient` 已具备 `answer/confirmPlan/approve` 且都带 `id`（回传同 `id`）；`useHitl` 订阅 `ask`/`show_plan`/`confirm_plan`/`approve` 与 `plan_progress` 事件。
- 纯 reducer `hitlMachine`（同 M9.3 `sessionMachine` 模式）：`queue: HitlRequest[]`；`ask`/`approve` 直接入队（带 id），`show_plan` 入队 **id=null**、`confirm_plan{id}` 补填最近空 id 计划 → 按钮才可点；`resolve_*` 按 id 出队。
- **协议要点**：daemon `show_plan`(bridge) **不带 id**，真正等待的是随后的 `confirm_plan{id}`——前端必须两步配对，否则模态无 id 无法回传。这与 M9 文档「`show_plan` 后由前端触发 `confirm_plan`」一致但注意 id 在第二条消息。

### plan 模态与 PLAN_PROGRESS 联动
- `show_plan` 携 `plan`(文本)/`plan_path`/`plan_steps[]({id,title,status})`；运行中 `PLAN_PROGRESS` 事件(`plan_update{step_id,status,note}`) 经 `useHitl` 的 `plan_progress` action 原地更新对应步骤状态，模态实时刷新，无需重开。

### 风险档配色映射（`ApprovalBadge.tsx`）
- `danger`→红(#c0392b)/`elevated`→橙(#b9770e)/`safe`→绿(#2e7d32)；`action.risk` 来自 daemon `approve` 的 `action{tool,risk,args,description,approval_request}`（`agent/runtime/approval.Action`）。`SandboxViz` 同构映射 `read-only`/`workspace-write`/`danger-full`。

### 阻塞性
- `useHitl.pending`(queue 长度>0) 驱动 `App.tsx` 输入框与发送按钮 `disabled`，模态关闭（resolve）后恢复。
