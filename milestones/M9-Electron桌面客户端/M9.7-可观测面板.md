# Step M9.7 可观测面板

## 实现方案

- **目标**：在桌面端提供可观测视图——上下文状态栏 + trace 树 + 日志 + 后台子 agent 状态，对齐 M8 状态栏与 M5 子 agent 状态。

- **协议现状与待扩展**：
  - **已有**（M7）：`usage{usage, text}`（上下文/token 用量）、`notify`（日志类消息）、`show_agents`/`show_skills`（列表）、`/bg` 后台任务（`session.list_background_tasks()`）。
  - **待扩展（daemon 侧，本期 M9.7 内落地）**：当前协议**无 trace 查询**。需新增：
    - C→S `trace.list{project_root, session_id?}` / `trace.get{project_root, trace_id}`；
    - S→C `trace_list{traces[]}` / `trace_tree{spans[]}`（span 含 `parent_id`，供构建父子树，对齐 `agent/obs/tracer.py` OTel 语义）。
    - 数据源：`<project_root>/.agent/traces.db`（`TraceStore`，M3 已落地），daemon 按 `project_root` 打开读取。

- **新增文件**（`desktop/src/features/obs/`）：
  - `StatusBar.tsx`：底部状态栏——上下文用量（来自 `usage`：prompt/completion tokens、窗口占比）、当前模式（plan/exec）、项目根、会话 id。
  - `TraceTree.tsx`：消费 `trace_tree`，按 `parent_id` 构建可折叠树（span 名/起止/耗时/状态）。
  - `LogView.tsx`：消费 `notify` 滚动日志（带级别/时间）；可清空。
  - `BackgroundAgents.tsx`：消费 `/bg`（`list_background_tasks`）展示后台子 agent 状态（运行中/完成 + task_id）；
    并订阅 `notify`（子 agent 完成通知）实时刷新（M5 后台 agent 完成自动 `notify`）。
  - `obsApi.ts`：封装 `trace.list`/`trace.get` 调用。

- **关键设计**：
  1. **上下文状态栏**：`usage` 消息（daemon 在 `step` 收尾 `transport.report_usage`）驱动；实时显示窗口占用百分比（与 M8 状态栏一致口径）。
  2. **trace 树**：daemon 新增 `trace.query` 类消息后，面板可「按会话查看 trace 树」；span 父子由 `parent_id` 重建（M3 已产出该关系）。
  3. **日志**：`notify` 实时追加；区分 agent 通知与错误。
  4. **后台子 agent**：`/bg` 初始拉取 + `notify` 增量；点击可跳转到对应会话/查看结果。
  5. **与 M9.0 协同**：所有 trace/obs 查询必须带 `project_root`（多项目隔离）。

- **依赖/环境**：M9.0 多项目 daemon、M9.2 `DaemonClient`（含新增 trace 消息订阅）、M3 `TraceStore` 读取。

## 知识沉淀

> 完成本步后填写：trace 查询消息最终形态、TraceStore 读取路径（按 project_root）、span 树重建算法。同步追加到 `knowledge/INDEX.md`。

- **trace 查询消息最终形态**：
  - C→S `trace.list{project_root, session_id?}`（带 `id` 配对）→ S→C `trace_list{project_root, traces[]}`，每项 `{session_id, span_count, first_ts, last_ts}`。
  - C→S `trace.get{project_root, trace_id}`（trace_id==session_id，带 `id`）→ S→C `trace_tree{session_id, spans[]}`，span 含 `parent_id` 供客户端重建父子树。
  - `MsgType` 新增 4 值（`trace.list`/`trace.get`/`trace_list`/`trace_tree`），共 32，契约检查 `scripts/check-msgtype.mjs` 仍一致。
- **TraceStore 读取路径（按 project_root）**：`start_daemon` 内 `trace_store_for(pr)` 惰性缓存 `TraceStore(_anchor_path(s.obs.db_path, pr))`，经 `SessionRegistry.trace_store_factory` 注入；`_route` 的 `_trace_list`/`_trace_get` 调 `registry._trace_store_factory(pr)`。`store.list_traces(session_id=None)` 按 `GROUP BY session_id` 聚合。多项目隔离：不同 `project_root` 落到各自 `.agent/traces.db`，互不可见（集成测试 `test_trace_isolation_across_projects` 验证）。
- **span 树重建算法**：`server._span_to_dict(s)` 序列化（含 `status='open'|'ok'`、`meta`、`logs`）；客户端 `features/obs/traceTree.ts` 的纯函数 `buildTree(spans)` 用 `span_id→node` Map + `parent_id` 挂子树（悬空 parent 视为根），`flattenTree` 供顺序遍历；已单测。
- **桌面端请求/响应配对**：`DaemonClient.requestResponse` 用 `crypto.randomUUID()` 生成 `id` + 一次性 `onMessage` 按 `env.id` 解 Promise；`listTraces`/`getTrace` 封装。面板组件见 `features/obs/`（ObsPanel/StatusBar/LogView/TraceTree/BackgroundAgents/obsApi/useObs）。
- 同步沉淀见 `knowledge/INDEX.md` 的 **M9.7 沉淀**。
