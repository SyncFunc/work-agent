# Step M7.4 session attach / switch / 回放缓冲

## 实现方案
- **目标**：前端 attach 会话并回放近期活动；切换 = detach + attach；后台子 agent 无人 attach 仍运行；**回放不重复渲染瞬时参数预览**。
- **改动文件**：`agent/daemon/registry.py`、`agent/daemon/bridge.py`、`agent/daemon/server.py`（消息路由 `replay_start` / `replay_end`）。
- **回放缓冲（修复点②，核心）**：`SessionHandle.event_buffer = deque(maxlen=K)`（建议 `K=200`，O(1) 追加与截断，防长期无人 attach 会话内存膨胀）。`BridgeTransport` 的 sink 收到事件时**仅当 `not ev.transient` 才写入缓冲**（`emit` 产生的 `tool_call_delta` 带 `transient=True`，只实时转发、不进缓冲）。`session.attach` / `session.switch` 时：先发 `replay_start`，再批量补发缓冲内最近 K 条持久化 `event`，最后发 `replay_end`；客户端据此重建画面，其后实时事件照常流入。
- **实时渲染不受影响**：客户端 sink 仍消费全部事件（含 `transient` 的 `tool_call_delta`），仅回放批次排除瞬时事件，避免参数预览重复出现。
- **后台子 agent 持续运行**：`Session._bg_tasks` 挂在 daemon 单循环，无人 `attach` 时仍推进；其事件经父 `Session` 的 `EventStream` sink 进入缓冲（仅持久化事件），attach 后由回放呈现近期活动。

## 验收标准
- [x] 新会话产生 N 条事件（含 `text` 增量与 `tool_call_delta` 瞬时）后 detach + switch 回，客户端收到 `replay_start` + 最近 K 条**持久化** `event` + `replay_end`，画面重建无空白、且 `tool_call_delta` 参数预览不重复出现。
- [x] 环形缓冲定长 O(1)，长期无人 attach 会话内存不膨胀。
- [x] 多会话并发：A、B 各自独立 step、事件互不串台；后台子 agent 无人 attach 持续运行，attach 后回放近期活动。

## 知识沉淀
> 设计期：回放缓冲与实时渲染是**两条消费路径**——实时消费全部事件（含 `transient`），回放仅消费持久化事件。这是修复「`tool_call_delta` 重复渲染」的关键区分。`transient` 标记是 core `Event` 的极小扩展（`emit` 置 `True`，`to_dict` 不计），已在 M7 设计期于 `agent/core/events.py` 落地。
