# Step M7.5 与 run / chat 共存 + 抽取 dispatch_command

## 实现方案
- **目标**：`run` / `chat` 进程内入口完全保留；抽共享命令分发避免协议路径重复实现命令语义；新增 `daemon` + `client` 并行入口。
- **改动文件**：`agent/cli.py`（新增 `daemon` / `client` 子命令 + 抽取命令分发）。core 零改动。
- **共存**：`run` / `chat` 进程内路径完全保留，单测不受影响。
- **抽取命令分发（对 `cli.py` 的低风险重构，非 core）**：把 `_chat_repl` 内的 `/context` `/compact` `/plan` `/exec` `/approve` `/skills` `/agents` `/skill` `/agent` `/bg` 分发逻辑抽为共享函数（如 `agent/core/session_command.py: dispatch_command(session, cmd, arg, transport) -> bool`），`_chat_repl` 与 daemon server 共用，避免协议路径重复实现命令语义。

## 验收标准
- [x] 既有 `run` / `chat` 进程内单测全绿，行为无回归。
- [x] `dispatch_command` 被 `_chat_repl` 与 daemon `command` 消息处理共用，命令语义唯一来源。
- [x] `agent daemon` 与 `agent client` 子命令可用；与 `run` / `chat` 互不干扰。
- [x] **不变量**：`agent/core/{loop,session,transport,events}.py` 无实质改动（仅 `Event` 新增 `transient` 字段与 `EventType` 枚举；`Session` 暴露只读 `event_stream`）。

## 知识沉淀
> 设计期：命令语义必须单一来源——抽 `dispatch_command` 后，进程内与 daemon 协议路径共用，杜绝「同命令两套实现」漂移。这是本里程碑唯一对 `cli.py` 的结构性重构（非 core）。

## 实施记录（✅ 已完成）
- 新增 `agent/core/session_command.py`：`dispatch_command(session, raw, transport, settings, *, feedback=None) -> bool`（异步，因 `/compact` 需 `await compact()`）。
  - 覆盖 `/plan` `/exec` `/approve` `/mode` `/context` `/compact` `/skills` `/agents` `/skill` `/agent` `/bg`；未知 slash 命令也返回 `True`（避免误当任务发往模型）。
  - `feedback` 回调：进程内用 `typer.echo(..., err=True)`，daemon 用 `transport.notify`，单一来源。
- `cli._chat_repl` 内联命令链替换为 `if await dispatch_command(...): continue`（进程内路径零重复）。
- daemon `server._command` 复用同一 `dispatch_command`（协议路径零重复）；`/switch` 在 server 层拦截为 `session.switch`。
- 验收：全量 `pytest -q` = 380 passed（含既有 run/chat 回归）；`test_session_command.py` 覆盖 dispatch 语义。
