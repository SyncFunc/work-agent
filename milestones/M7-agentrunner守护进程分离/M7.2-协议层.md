# Step M7.2 协议层（protocol + BridgeTransport + HITL future）

## 实现方案
- **目标**：定义 WebSocket 消息信封与编解码；服务端 `BridgeTransport(AgentTransport)` 把 HITL 经带 `id` 的请求 / 应答闭环，事件经 sink 实时转发；确立完整协议规范。
- **改动文件（新增）**：`agent/daemon/protocol.py`、`agent/daemon/bridge.py`。`protocol.py` client / server 共用，便于单测往返。
- **3.1 传输**：WebSocket `ws://127.0.0.1:<port>`（`port` 取自 `settings.daemon.port`，默认 `18789`）。仅绑回环地址。本地 HTTP 同端口或独立端口提供 `/health` 与未来 Web 静态资源；Web 与 CLI 客户端共用同一 WS 端点。依赖：实现时向 `pyproject.toml` 增加 `websockets`（仅 daemon / client 路径 import）。
- **3.2 消息信封（JSON）**：`{ "type": "<str>", "id"?: "<str>", "session"?: "<str>", "payload": { ... } }`。`id` 仅用于请求 / 应答配对（HITL），服务端生成（`uuid4().hex`），客户端原样回传。`session` 多数消息带；`hello` / `session.new` / `session.list` 除外。
- **3.3 消息类型枚举**：
  - **Client → Server**：`hello`{client_type,version}、`session.new`{name?}、`session.attach`{session_id}、`session.switch`{session_id}、`session.detach`{}、`session.list`{}、`task.send`{text, yes?, plan?}、`answer`{id, text}、`confirm_plan`{id, confirmed}、`approve`{id, approved}、`command`{name, args}（`/context`/`/compact`/`/plan`/`/exec`/`/approve`/`/skills`/`/agents`/`/agent`/`/bg`/`/sessions`/`/switch`/`/mode`）。
  - **Server → Client**：`welcome`{daemon_version, protocol_version}、`session.created`{session_id, name}、`attached`{session_id}、`detached`{session_id}、`session_list`{sessions:[{id,name,attached,running,last_activity}]}、`event`{event: Event.to_dict()}、`replay_start`/ `replay_end`{}、`ask`{id, question: Question.to_dict()}、`confirm_plan`{id}、`approve`{id, action: Action.to_dict()}、`show_plan`{plan, plan_path, plan_steps}、`show_questions`{questions}、`show_skills`{specs}、`show_agents`{specs}、`notify`{message}、`usage`{usage}、`close`{text?}、`error`{code, message}。
- **3.4 与 `AgentTransport` 的双向映射**：服务端 `BridgeTransport` 把每个协议方法映射到一条出站消息，并把对应应答消息映射回方法返回值：

  | `AgentTransport` 方法 | 服务端动作 | 客户端动作 | 回传 |
  |---|---|---|---|
  | `ask(question)` | 发 `ask`{id, question.to_dict()}；`await future` | 渲染并 `prompt_toolkit` 提问 | `answer`{id, text} |
  | `show_questions(qs)` | 发 `show_questions`{qs} | 渲染面板 | 无 |
  | `show_plan(res)` | 发 `show_plan`{plan, plan_path, plan_steps} | 渲染面板 | 无 |
  | `confirm_plan()` | 发 `confirm_plan`{id}；`await future` | 渲染并确认 | `confirm_plan`{id, bool} |
  | `approve(action)` | 发 `approve`{id, action.to_dict()}；`await future` | 渲染并确认 | `approve`{id, bool} |
  | `notify(msg)` | 发 `notify`{msg} | 缓冲后刷出 | 无 |
  | `show_skills(specs)` | 发 `show_skills`{specs} | 渲染 Table | 无 |
  | `show_agents(specs)` | 发 `show_agents`{specs} | 渲染 Table | 无 |
  | `bind(stream)` | `stream.subscribe(sink)`；sink 把 `Event.to_dict()` 发 `event` | 收到 `event` → `Event.from_dict()` → 渲染 | 无 |
  | `close()` | 发 `close` | flush 渲染 | 无 |
  | `report_usage(usage, answer)` | 发 `usage`{usage 或近似} | 渲染 token 面板 | 无 |

  `interactive` 属性返回 `True`（HITL 由远端客户端承接），因此 `Session.step` 正常走 `ask` / `confirm_plan` / `approve` 分支，无需改 core。
- **3.5 事件衔接与 HITL future 机制**：实时 `event` 消息承载 `Event.to_dict()`；`BridgeTransport` 持 `pending: dict[id, asyncio.Future]`，方法发送请求时 `loop.create_future()` 存入，收到对应 `id` 应答后 `future.set_result(...)` 唤醒 `await` 的 `Session.step` 协程。请求 `id` 由 daemon 事件循环生成、跨连接唯一。回放缓冲仅收非 `transient` 事件（见 M7.4）。

## 知识沉淀
> 设计期：协议是 `AgentTransport` 的**序列化映射**，不是新协议；事件复用 `Event.to_dict/from_dict` 是天然序列化边界。HITL 用请求 `id` + `Future` 配对，零阻塞跨进程。新增实时渲染一律走 `event`，不回退 loop 加回调。`transient` 标记（core 极小扩展）使回放可排除瞬时事件。

## 实施记录（✅ 已完成）
- 新增 `agent/daemon/bridge.py`：`BridgeTransport(AgentTransport)`。`ask`/`confirm_plan`/`approve` 发送带 `id` 的请求并 `await` 一个 `asyncio.Future`；`resolve(id, value)` 由 server 收到客户端应答时唤醒；`pending: dict[id, Future]` 配对、`interactive == True`。
- `bind(stream)` 订阅事件：实时经 `event` 消息转发；同时写入 `SessionHandle.event_buffer`（仅非 `transient`，见 M7.4 修复点②）。`flush()` 等待在飞事件转发完成，保证 `FINAL` 等事件先于 `CLOSE` 落地。
- 协议规范 §3.1–3.5 全部落地：`MsgType` 枚举覆盖 C→S / S→C 全部类型；`Connection.send` 出站加锁保证顺序。
- 验收：单测 `test_daemon.py`（真实 websockets 临时端口）覆盖 `hello`→`welcome`、`session.new`→`created`+`attached`、`session.list`、HITL 经 `answer` 唤醒并 `close`、并发 `task.send` 经 `busy` 串行化，全绿。
