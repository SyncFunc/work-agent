# Step M2.4 工具与循环集成（bash→Sandbox；loop 接 gate；session 建 gate）

## 实现方案

- 目标：把 M2.1 的 `SandboxExecutor` 与 M2.2 的 `ApprovalGate` 接到执行链路。
- 改动文件：
  - `agent/tools/bash.py`：`bash` 工具改为构造 `ExecRequest` 并经 `get_executor()` 执行。
  - `agent/core/loop.py`：`AgentLoop.__init__` 增可选 `sandbox=None, gate=None`；`_exec_tools` 执行前过 `ApprovalGate`。
  - `agent/core/session.py`：从 `Settings` 构建 `ApprovalGate` 与 `build_executor`。
- **M2 重构后简化**：去掉 deny/escalated 分支、去掉 enable_elevation 判断、`allow`→`exec_policy`、`UNTRUSTED`→`UNLESS_TRUSTED`。
- 关键接口/算法：

  ```python
  # loop.py — 精简后的审批门集成
  if self.gate is not None:
      action = Action(
          tool=tc.name,
          risk=spec.risk,
          args=tc.arguments or {},
          description=self._describe(tc),
          approval_request=tc.approval_request,
      )
      d = self.gate.decide(action)
      if d.verdict == "ask":
          ok = await self.gate.authorize(action)
          if not ok:
              return ToolResult(ok=False, error="rejected by user approval")
          elevated = d.elevated_profile  # 自动提权
      # verdict == "allow"：不提权
  ```

- **提权自动**：不再需要 `enable_elevation` 开关。只要命令经 ASK→批准且需联网，自动以 `elevated_profile` 临时执行。
- **`on-failure` 模式**：先 ALLOW 执行；若 `res.ok is False` 且 `mode==ON_FAILURE`，再 `await transport.approve` 问补救方案。
- **`Session` 构建**：`ApprovalGate` 传 `exec_policy`（不再传 `allow`/`deny`）。
- **PLAN 模式复用 sandbox+approval**：plan mode 不再有独立的风险门控（`_risk_blocked`/`plan_mode_bash_allow` 已移除），执行限制由 `sandbox_profile` + `approval_mode` 自然保证（如 `read-only` + `never`）。plan mode 只负责控制 `present_plan` 工具和计划确认流程。

## 知识沉淀

- **loop 注入点**：`AgentLoop(model, registry, settings, sandbox=None, gate=None)`。
- **提权自动**：不再需要 `enable_elevation` 开关和 `approval_elevate_on_approve` 配置。
- **`exec_policy`**：`ApprovalGate` 构造函数参数改为 `exec_policy`（替代 `allow`），仅 `unless-trusted` 有效。
- **系统提示**：`system.md` 中的 `sandbox_allow` → `sandbox_exec_policy`，去掉 `sandbox_deny` 和 `elevate_on_approve`。
