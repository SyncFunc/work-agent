# Step M2.4 工具与循环集成（bash→Sandbox；loop 接 gate；session 建 gate）

## 实现方案

- 目标：把 M2.1 的 `SandboxExecutor` 与 M2.2 的 `ApprovalGate` 真正接到执行链路——`bash` 走沙箱执行；`AgentLoop` 在执行每个工具前过审批门；`Session` 按 `Settings` 构建 gate 与 executor 并接入 HITL。
- 改动文件：
  - `agent/tools/bash.py`：`bash` 工具改为构造 `ExecRequest` 并经 `get_executor()` 执行，把 `ExecResult` 转回 `ToolResult`（超长截断仍走 `_cap_result`）。
  - `agent/core/loop.py`：`AgentLoop.__init__` 增可选 `sandbox=None, gate=None`；`_exec_tools` 在执行前构造 `Action` 并 `await gate.authorize`（无 gate 则跳过）；`bash` 经 `sandbox.run`，其余经 `spec.fn`。
  - `agent/core/session.py`：从 `Settings` 构建 `ApprovalGate`（含 `ui` 与 `noninteractive_default`）与 `build_executor`，传给 `AgentLoop`。
- 关键接口/算法（设计文档 §4.1）：

  ```python
  # loop.py
  class AgentLoop:
      def __init__(self, model, registry, settings, tracer=None,
                   sandbox=None, gate=None):   # 新增可选依赖，默认 None=不隔离/不审批
          ...
      async def _exec_tools(self, calls, stream, agent_span):
          for tc in calls:
              spec = registry.get(tc.name)     # UnknownTool 降级（既有）
              action = Action(tool=tc.name, risk=spec.risk,
                              args=tc.arguments, description=_describe(tc),
                              approval_request=getattr(tc, "approval_request", False),
                              escalated=_is_escalated(spec, tc))   # 模型可在单条命令标 approval_request（on-request 问此条）；_is_escalated 检测装包/改系统配置等提权
              if self.gate is not None:
                  ok = await self.gate.authorize(action)
                  if not ok:
                      res = ToolResult(ok=False, error="blocked/rejected by approval")
                      # 落事件流 + 回填，不崩循环（既有降级机制）
                      ...
                      continue
              if tc.name == "bash" and self.sandbox is not None:
                  res = await self._run_bash_in_sandbox(tc.arguments)
              else:
                  res = await spec.fn(tc.arguments)   # read/write/edit 直接执行
              ...  # 既有的 tool_call_id 配对 + EventStream 记录不变
  ```

  - `on-failure` 模式：先 ALLOW 执行（不事前问）；若 `res.ok is False` 且 `gate.mode == ON_FAILURE`，再 `await transport.approve(action)` 询问**补救方案**（重试/换参数/跳过）——注意是「失败后问怎么补救」，不是「事前问能不能跑」；仍拒则维持 `res`。
- **`bash` 工具改造**（`bash.py`）：

  ```python
  from agent.runtime.sandbox import get_executor, SandboxProfile
  async def bash(args):
      profile = SandboxProfile(settings_sandbox_profile)   # 经模块级读取
      req = ExecRequest(cmd=args["cmd"], cwd=Path.cwd(), env=..., timeout=..., profile=profile)
      r = await get_executor().run(req)                    # 沙箱执行
      out = r.output
      if r.error:
          out = out + ... + f"[stderr]\n{r.error}"
      # 仍经 ToolRegistry.run 的 _cap_result 截断（在 registry 层统一做）
      return ToolResult(ok=r.ok, output=out, error=None if r.ok else f"exit code {r.returncode}")
  ```

  - 注意：`bash.py` 不应直接 import `settings.load_settings()` 每次调用（性能/递归）；改为 `get_executor()` 模块级缓存当前执行器，`sandbox_profile` 通过 `get_executor` 构造时注入（executor 已持有 profile）。`bash` 只需 `req.profile = executor.default_profile`。
- **`Session` 构建**（设计文档 §4.1）：`Session.step` 已接收 `transport: AgentTransport`（M1.6 重构后，`SessionUI`/`LoopPresenter` 已合并为统一 `AgentTransport`）。`Session.step` 构建 `AgentLoop` 时传入：
  - `sandbox = build_executor(settings.sandbox_mode, workspace=..., profile=...)`；
  - `gate = ApprovalGate(settings.approval_mode, allow=settings.approval_allow, deny=settings.approval_deny, ui=transport, noninteractive_default=settings.noninteractive_default, sandbox_profile=settings.sandbox_profile)`（`ui=transport`：`AgentTransport` 结构满足 `ApprovalUI` 窄协议）。
- **不破坏 PLAN**：PLAN 模式 `_risk_blocked` 逻辑完全不动；`ApprovalGate` 仅在 EXEC 模式（`plan_mode=False`）由 `Session` 注入；PLAN 模式 `Session` 可不传 gate（或传但 PLAN 门控已先拦）。确认 `test_plan.py` 不回归。
- 依赖/环境：复用 M2.1/M2.2 产物；无新依赖。

## 知识沉淀

> 完成本步后填写：loop 注入点、bash 改造要点、session 构建、PLAN 边界。同步追加到 `knowledge/INDEX.md`。

- **loop 注入点**：`AgentLoop(model, registry, settings, tracer=None, sandbox=None, gate=None)`；`sandbox=None`/`gate=None` 时退化为 M1 行为（零回归）。`_exec_tools` 先 `gate.authorize` 再执行；`bash`+`sandbox` 走 `executor.run`，其余走 `spec.fn`。
- **bash 改造铁律**：`bash` 经 `get_executor()`，**不直接 `subprocess`**；`get_executor()` 模块级缓存，避免每次调 `load_settings`。`profile` 由 executor 持有，`bash` 只取 `executor.default_profile` 填 `ExecRequest`。
- **Session 构建**：`Session.step` 从 `Settings` 构建 `ApprovalGate(ui=transport, ...)`（传入的 `AgentTransport`）与 `build_executor(...)` 注入 `AgentLoop`；PLAN 模式可选择性不传 gate（PLAN 门控先拦）。
- **不破坏 PLAN**：`loop._risk_blocked` 不动；`ApprovalGate` 仅 EXEC 模式生效（设计文档 §1 正交约定）。`test_plan.py` 必须保持绿。
- **on-failure**：先 ALLOW 执行（不事前问），`res.ok is False` 且 `mode==ON_FAILURE` 时再 `await transport.approve` 问**补救方案**（重试/换参数/跳过）；是「失败后问补救」而非「重问同一条」。
- **对 M2.5 约束**：`transport.approve(action)` 由 CLI 的 `TerminalTransport` 实现（M2.5）；统一协议 `AgentTransport` 需新增 `async approve` 方法（gate 经窄协议 `ApprovalUI` 消费）。
