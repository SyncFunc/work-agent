# Step M2.3 分层权限配置（Settings）

## 实现方案

- 目标：把沙箱与审批的**所有可变策略**落进声明式 `Settings`（YAML + env + CLI），代码零硬编码策略，与既有分层配置（CLI > env > 项目 YAML > 用户 YAML > 内置默认）一致。
- 改动文件：`agent/config/settings.py`（`Settings` 增字段）、`agent/config/settings.example.yaml`（模板）、`<project>/.agent/settings.yaml`（用户已存在，文档说明新增键）、`agent/runtime/__init__.py` 无改动。
- 新增字段（设计文档 §5 / §3）：

  ```python
  # agent/config/settings.py → class Settings
  # 沙箱
  sandbox_mode: str = "local"            # local | docker | external
  sandbox_profile: str = "workspace-write"   # read-only | workspace-write | danger-full
  sandbox_image: str = "alpine:latest"   # docker 模式用的镜像
  sandbox_danger_allow_bash: bool = False    # 是否允许 danger-full 放行 bash（默认关，需显式开）

  # 审批（Codex AskForApproval 四模式）
  approval_mode: str = "untrusted"       # untrusted | on-request | on-failure | never
  approval_deny: list[str] = [           # 永远拦截的命令前缀/正则（安全网）
      "rm -rf /", "mkfs", "dd if=", ":(){",  # 经典毁灭性命令前缀（示例，可覆盖）
  ]
  approval_allow: list[str] = [          # 短路 ALLOW，跳过确认疲劳（示例）
      "ls", "cat", "echo", "git status", "git diff",
  ]
  noninteractive_default: str = "allow"  # 非交互（run/测试）无 HITL 时的默认 verdict
  ```

- **枚举校验**：`sandbox_mode`/`sandbox_profile`/`approval_mode` 在 `settings.py` 用轻量校验（启动时或字段 validator）确保取值合法，非法值给清晰报错而非静默。可用 `__post_init__` 或 pydantic `field_validator`（当前 `Settings` 是 pydantic-settings，可加 `field_validator`）。
- **列表覆盖语义**：`approval_deny`/`approval_allow` 是 pydantic `list`，**项目级 YAML 整体覆盖**默认（沿用 M1.2 的 `plan_mode_bash_allow` 同样约定——别误删默认项；文档示例给出完整列表供复制）。
- **env 映射**：可用 `AGENT_SANDBOX_MODE` / `AGENT_APPROVAL_MODE` 等环境变量覆盖（pydantic-settings 默认把字段名映射到 `AGENT_<FIELD>` 大写，需确认命名；或保持 `LLM_` 风格的前缀约定）。
- **CLI 覆盖**：`agent run/chat` 增加 `--sandbox-mode` / `--sandbox-profile` / `--approval-mode` 选项（M2.5 接），经 `load_settings(**overrides)` 注入（既有机制）。
- 依赖/环境：无新依赖；改动向后兼容（新字段均有默认值）。

## 验收标准

- [ ] 命令/测试：`pytest` 已有 `tests/test_settings.py`（或新增用例）断言：默认 `sandbox_mode=local`/`sandbox_profile=workspace-write`/`approval_mode=untrusted`；项目级 YAML 覆盖生效（monkeypatch `AGENT_PROJECT_ROOT` 指向含覆盖 YAML 的 tmp 目录）；非法 `approval_mode=bogus` 在 `load_settings` 时抛清晰错误。
- [ ] 行为：`approval_deny` 默认值含示例毁灭性前缀；项目 YAML 写 `approval_deny` 整体覆盖（不追加）——测试验证覆盖语义。
- [ ] 不变量：所有新字段有默认值，旧配置/旧测试（不写这些字段）仍正常加载；与 M1 四级合并顺序一致（CLI > env > 项目 YAML > 用户 YAML > 内置默认）。

## 知识沉淀

> 完成本步后填写：字段清单、枚举校验位置、列表覆盖语义、env 命名。同步追加到 `knowledge/INDEX.md`。

- **字段清单**：`sandbox_mode`/`sandbox_profile`/`sandbox_image`/`sandbox_danger_allow_bash`；`approval_mode`/`approval_deny`/`approval_allow`/`noninteractive_default`（均带默认值，向后兼容）。
- **枚举校验**：在 `Settings` 用 `field_validator` 校验三个枚举字段；非法值启动即报错（不要静默落默认）。
- **列表覆盖语义（易忘）**：`approval_deny`/`approval_allow` 是 pydantic `list`，项目级 YAML 写会**整体覆盖**默认值（非追加），别误删默认项；文档/模板给出完整默认列表供复制。
- **env 命名**：确认 pydantic-settings 的 env 映射（可能 `AGENT_SANDBOX_MODE` 等）；CLI 用 `load_settings(**overrides)` 注入，优先级最高。
- **对 M2.4 约束**：`session.py` 从 `Settings` 读上述字段构建 `ApprovalGate(...)` 与 `build_executor(...)`；`bash` 工具读 `sandbox_profile` 构造 `ExecRequest`。
