# Step M8.3 HITL 模态

## 实现方案

- 目标：HITL（`ask` / `approve` / `confirm_plan`）用 `ModalScreen` + 线程安全 `Future` 实现，
  覆盖单选 / 多选 / 审批 / 计划确认；在 worker 线程被 `await`，用户操作经 `loop.call_soon_threadsafe`
  设置 `Future` 唤醒（Textual 官方跨线程交互模式，调研 §4.2）。
- 改动文件：
  - 新建 `agent/tui/screens.py`：
    - `AskScreen(ModalScreen)`：渲染问题 + 选项；按键 / 点击返回选择 → `self.dismiss(result)`。
    - `ApproveScreen(ModalScreen)`：y/N（及 always / details）→ `dismiss(bool)`。
    - `PlanScreen(ModalScreen)`：展示计划 → 确认 / 拒绝。
  - `agent/runtime/textual_transport.py`（完善 M8.1 占位）：
    - `ask(self, question, options) -> str`：在 worker 线程创建 `Future`；
      `app.call_from_thread(app.push_screen, AskScreen(..., on_result=fut.set_result))`；`return await future`。
    - `approve` / `confirm_plan` 同理。
    - 屏幕的 `on_result` 经 `app.loop.call_soon_threadsafe(fut.set_result, value)` 唤醒
      （worker loop 与 app loop 不同，必须线程安全唤醒）。
- 关键接口/算法：
  - `Future` 用 `concurrent.futures.Future`；唤醒必须线程安全（Textual 官方模式）。
  - `push_screen` 经 `call_from_thread` 从 worker 线程切到 app 主线程。
- 依赖/环境：M8.2 的 worker 已建；`ModalScreen` 内置。

## 知识沉淀

### ① ModalScreen + 线程安全 Future 标准范式（核心）

- worker 线程（M8.2 的 session driver）触发 `ask/approve/confirm_plan` 时：
  1. 创建 `concurrent.futures.Future`（跨线程 `set_result` 线程安全）；
  2. `self._bridge(self._app.push_screen, XxxScreen(..., future))` —— 经 `_bridge` 把推屏投递回 app 主线程；
  3. `return await asyncio.wrap_future(fut)` —— 把 `concurrent.futures.Future` 挂回 worker 自己的事件循环（**与 app 主线程是两个不同 loop**）。
- 屏幕（主线程）用户操作后：`_finish(value)` 里 `if not self._future.done(): self._future.set_result(value)` 再 `self.dismiss()`，唤醒被 `await` 的 worker step。

### ② 三个 HITL 屏幕的入参 / 出参契约

| 屏幕 | 入参 | 出参（Future 值） | 交互 |
|---|---|---|---|
| `AskScreen` | `question: Question`, `future` | `str` | 有 `options` 走数字键选择（多选空格切换、回车确认）；无 `options` 走 `Input` 自由输入、回车提交；Esc→`""` |
| `ApproveScreen` | `action: Action`, `future` | `bool` | `y/Y`→True，`n/N`/Esc→False |
| `PlanScreen` | `future` | `bool` | 计划正文已由 `show_plan` 渲染到主消息区，这里只做确认：`y/Y`→True，`n/N`/Esc→False |

- `Question` 字段：`question: str`、`options: list[str]|None`、`multiSelect: bool`（`agent/core/intent.py`）。
- `Action` 字段：`tool/risk/args/description/approval_request`（`agent/runtime/approval.py`）；屏幕只读展示，不回调。

### ③ worker loop ≠ app loop —— 跨线程唤醒要点

- `_bridge(target, *args)` 线程感知：若 `threading.get_ident() == app._thread_id`（run_test 同线程）直接 `target(*args)`；否则 `app.call_from_thread(target, *args)`（Textual 官方跨线程 API）。
- 测试场景（`run_test`）里测试协程与 app 同线程 → 走直调分支，`push_screen` 同步生效，可直接 `pilot.press` 交互；真实场景 worker 异线程 → 走 `call_from_thread`，屏幕在主线程 `set_result` 经 `asyncio.wrap_future` 唤醒 worker。

### ④ 与 `TerminalTransport` HITL 对照（行为一致）

- `TerminalTransport`：交互式用 `input()` 同步读；非交互式把问题缓冲到 `_pending_notifications`，由 `flush_notifications()/close()` 刷出。
- `TextualTransport`：交互式用 `ModalScreen + Future` 异步收集；语义契约完全相同（都 `await transport.ask/approve/confirm_plan`），**协议 / loop / session 零改动**。

### ⑤ 两个踩坑（M8.3 实测）

1. **审批门触发条件**：`ApprovalGate.decide` 对 `risk=="read"` 有「只读自动放行」早返回分支，会**先于** `approval_request` 判定生效。测试触发审批必须注册 `risk="exec"/"edit"` 的工具（如 `_make_approval_session` 用 `tool("echo", risk="exec")`），否则 `echo`(read) 永不弹审批框。
2. **循环导入**：`textual_transport.py` 若在顶部 `from agent.tui.screens import ...`，会在特定导入顺序下触发 `agent.tui.__init__` → `agent.tui.app` → `textual_transport`（部分初始化）的 `ImportError: cannot import name 'TextualTransport'`。已改为在 `ask/approve/confirm_plan` 三个方法内**惰性导入** `agent.tui.screens`，打破循环。
3. **`@mount` 竞态**：worker 线程高频经 `call_from_thread` 调 `_mount` 时，反复 `self.query_one("#log")` 偶发 `NoMatches('#log')`（屏幕就绪竞态）。已在 `on_mount` 缓存 `self._log = self.query_one("#log", VerticalScroll)`，此后只 `self._log.mount(...)`，稳定且更高效。
