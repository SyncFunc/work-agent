# Step M8.1 TextualTransport 事件桥接

## 实现方案

- 目标：实现 `TextualTransport(AgentTransport)`，订阅 `EventStream`，把核心事件映射为 UI 部件更新
  （经 `app.call_from_thread`）。用 `FakeModel` + `run_test` 验证消息 / 工具块出现。
- 改动文件：
  - 新建 `agent/runtime/textual_transport.py`：`class TextualTransport(AgentTransport)`。
    - `bind(self, stream)`：保存 stream 引用并 `stream.subscribe(self.on_event)`（订阅式，对齐 `TerminalTransport`）。
    - `on_event(self, ev)`：**公开方法**（对齐 `TerminalTransport._on_event`，便于 daemon `client` 未来复用，见调研 §4.6）；
      按 `ev.type` 分派：
      - `TEXT` / `ASSISTANT_MESSAGE` → `app.call_from_thread(self._append_assistant, ev.text)`
      - `TOOL_USE` → `app.call_from_thread(self._append_tool_block, ev)`
      - `TOOL_RESULT` → `app.call_from_thread(self._update_tool_block, ev)`
      - `PLAN_PROGRESS` → `app.call_from_thread(self._append_plan, ev)`
      - `DECISION` → `app.call_from_thread(self._append_decision, ev)`
    - `ask` / `approve` / `confirm_plan` 暂 `raise NotImplementedError`（M8.3 完善）；
      `show_*` / `notify` / `report_usage` 先落到 `RichLog` / `Footer`；`close()` 解除订阅。
  - `agent/tui/app.py`：加 `append_*` 回调方法（写入 `RichLog`）；`ChatApp` 持有 `transport` 引用。
- 关键接口/算法：
  - `TextualTransport` 必须与 `TerminalTransport` 同签名实现 `AgentTransport`
    （见 `agent/core/transport.py` 协议）；HITL 方法暂留待 M8.3。
  - 事件 → 部件映射函数清单（如上），每个回调只做 UI 写入，不含 loop 逻辑。
- 依赖/环境：需先有 M8.0 的 `ChatApp`；`FakeModel`（已有）用于 `run_test` 驱动 session。

## 验收标准

- [ ] `test_transport_maps_events`：构造 `ChatApp` + `TextualTransport`，订阅内存 `EventStream`，
  手动 `append(TEXT/TOOL_USE/TOOL_RESULT/DECISION)`，断言 `RichLog` 出现对应部件文本。
- [ ] 用 `FakeModel` 跑一次 `session.step` 经 `TextualTransport`，`run_test` 下主区出现助理消息与工具块。
- [ ] 不变量：`AgentTransport` 协议未变；`EventStream.append/emit` 未变；
  `on_event` 公开暴露（供 daemon `client` 未来复用）。

## 知识沉淀

- **`TextualTransport` 实现要点**：`__init__(app, *, context_mgr=None)` 持有 `ChatApp` 引用；
  实现 `AgentTransport` 全部方法。`bind(stream)` 订阅 `on_event` 并重置 `_tc_by_id`；`close()` 退订。
  `on_event` 是**公开方法**（对齐 `TerminalTransport._on_event`），便于 daemon `client` 复用。
- **事件类型 → 部件映射**（在 `on_event` 内）：
  - `TEXT` → `app.append_text(text, kind)`（kind=reasoning/content）
  - `TOOL_USE` → 记录 `_tc_by_id[id]`，`app.append_tool_use(tc)`
  - `TOOL_CALL_DELTA` → 瞬时事件，M8.1 忽略（M8.4 可接预览）
  - `TOOL_RESULT` → 按 `tool_call_id` 取 `tc`，`app.update_tool_result(tc, res)`
  - `PLAN_PROGRESS` → `app.append_plan_progress(ev)`
  - `USER` → `app.append_user(text)`
  - `DECISION` → `app.finalize_stream()`（定稿流式块）
- **跨线程桥接 `_bridge`**：真实场景 session 在 worker 线程、`app` 在主线程 → 用
  `app.call_from_thread`；但 **`run_test` 下测试与 app 同线程**，`call_from_thread` 会抛
  "must run in a different thread"，故 `_bridge` 用 `threading.get_ident() == app._thread_id`
  判定：同线程直接调用，跨线程才 `call_from_thread`。这是 Textual headless 测试的关键坑。
- **主消息区用 `VerticalScroll`**（而非 `RichLog`）：便于 `mount` 自定义部件并增量 `update`
  流式块（RichLog 的 `write` 只接静态 Rich renderable，不便更新）。`#log` 的 `id` 不变，M8.0 测试仍通过。
- **不变量已守住**：`AgentTransport` 协议 / `EventStream` / loop 未改；`on_event` 公开暴露。
- **踩坑**：① `rich.group` 在 rich 15 不存在 → 改 `from rich.console import Group`；
  ② `ToolResult` 在 `agent.runtime.registry`（非 `agent.core.model`）。
