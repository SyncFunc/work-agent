# Step M6.2 会话恢复流程（+ M7 冷启动衔接 + fork 恢复）

## 实现方案
- **目标**：从 sqlite 恢复完整会话（事件流 + messages + 计划 + 记忆摘要），含**中断检测**与 **fork 分支恢复**；daemon 重启后（冷启动）attach 旧 `session_id` 能从 sqlite 加载而非仅内存。
- **改动文件**：
  - 新增 `agent/context/session_recovery.py`：`rebuild_messages(stream)` + `detect_interruption(stream)`。
  - 改 `agent/core/session.py`：新增 `Session.from_store(session_id, store, ...)` 类方法或 `restore_session` 工厂——`load` events → 设 `self.event_stream` → `rebuild_messages` → 注入 `self.messages`。
  - 改 `agent/daemon/registry.py`：`SessionRegistry.get/new` 对「已存在于 SessionStore 的 session_id」走 `restore_session`（冷启动加载）；`event_buffer` 用加载 events 的最近 K 条播种。
  - 改 `agent/cli.py`：`resume <session_id>` 调 `restore_session`；新增 `fork <session_id>` 子命令 + REPL `/fork` 命令（调 `SessionStore.fork` 后 `restore_session`）。

- **与 M7 衔接**：热切换仍走内存 `event_buffer` 回放；**冷启动**（daemon 重启后 attach 旧 `session_id`）走 `restore_session` 从 sqlite 加载初始 events 进 `EventStream` 与 `event_buffer`。
- **依赖/环境**：复用 M6.1 `SessionStore`；记忆摘要（`summary.md`）由 `SessionMemory` 独立加载（已在 M4.4 落盘），恢复时随 `Session` 构造重新挂载。

## 验收标准
- [ ] `restore_session(session_id)` 重建的 `messages` 与崩溃前（完整重放）一致；`tool_calls` 必有对应 `tool_result`，无悬空。
- [ ] 中断场景：末条 `TOOL_USE` 无 `TOOL_RESULT` 时，重建 `messages` 不出现悬空 `tool_calls`，且含「继续」user 提示；下一轮 `step` 能正常续跑。
- [ ] 冷启动：daemon 重启后 `resume <id>`（同 id）从 sqlite 恢复，事件流与 `event_buffer` 播种正确，无需原进程在跑。
- [ ] fork 会话恢复：父与子各自 `resume` 均正确重建，子含父前缀事件，互不串台。
- [ ] 不变量（M4 双轨）：恢复后 `EventStream` 为完整未压缩序列；压缩由运行时 `ContextManager` 下一轮重算，不存压缩态。
