# Step M6.2 会话恢复流程（+ M7 冷启动衔接 + fork 恢复）

## 实现方案
- **目标**：从 sqlite 恢复完整会话（事件流 + messages + 计划 + 记忆摘要），含**中断检测**与 **fork 分支恢复**；daemon 重启后（冷启动）attach 旧 `session_id` 能从 sqlite 加载而非仅内存。
- **改动文件**：
  - 新增 `agent/context/session_recovery.py`：`rebuild_messages(stream)` + `detect_interruption(stream)`。
  - 改 `agent/core/session.py`：新增 `Session.from_store(session_id, store, ...)` 类方法或 `restore_session` 工厂——`load` events → 设 `self.event_stream` → `rebuild_messages` → 注入 `self.messages`。
  - 改 `agent/daemon/registry.py`：`SessionRegistry.get/new` 对「已存在于 SessionStore 的 session_id」走 `restore_session`（冷启动加载）；`event_buffer` 用加载 events 的最近 K 条播种。
  - 改 `agent/cli.py`：`resume <session_id>` 调 `restore_session`；新增 `fork <session_id>` 子命令 + REPL `/fork` 命令（调 `SessionStore.fork` 后 `restore_session`）。

## 知识沉淀
> M6.2 已完成（2026-07-22）。关键决策与约定：

- **事件→Message 映射规则**（`rebuild_messages`）：
  - `USER` → `user(content=task)`；`DECISION` → `assistant(text + tool_calls)`（每轮决策唯一真相来源）；`TOOL_RESULT` → `tool(content, tool_call_id)`。
  - `CLARIFY` / `PLAN` 事件：为上一轮 `DECISION` 中未配对的澄清/计划 tool_calls **补一条合成 tool 消息**（内容用原 loop 的占位文案），以确保 OpenAI 配对合法。
  - `FINAL` / `TEXT` / `TOOL_USE` / `PLAN_PROGRESS` / `ERROR` **忽略**：FINAL/TEXT 信息已含于 DECISION（DECISION 携带完整 `text`，FINAL 是重复收尾）；TOOL_USE 与 TOOL_RESULT 配对，重建只需结果侧。
- **中断检测 + 注入策略**：`detect_interruption` 用「存在 `TOOL_USE` 且无对应 `TOOL_RESULT`」判定（与 M7.4 回放缓冲「仅收非 transient」一致，瞬时 `tool_call_delta` 不参与）。`rebuild_messages` 在建完消息后若仍有 pending tool_calls（末轮决策缺配对结果），丢弃该悬空 `assistant` 消息并（默认）注入一条 user「（会话在上次中断处恢复，请继续执行未完成的工具调用）」，保证 messages 始终合法、可续跑。
- **完整可重放转录的关键改动**：新增 `EventType.USER`，`loop.run` 在每轮开头（conv 追加 user 消息后）`stream.append(Event(type=USER, text=task))`。这使 EventStream 包含用户轮次，重建 messages 才能与崩溃前 `self.messages` 逐条一致。终端/daemon 传输层未订阅 USER（走既有 `if/elif` 忽略分支），仅 `SessionStoreSink` 与 `BridgeTransport.event_buffer` 收集——不影响渲染。
- **`session_id` 冷启动恢复路径**：`Session.from_store(cls, model, reg, settings, store, session_id, ...)` 类方法 = `store.load` → 设 `self.event_stream` → `rebuild_messages` → 注入 `self.messages`；`__init__` 的 `store.create` 幂等（INSERT OR IGNORE + UPDATE updated_at），不破坏既有元数据行。
- **与 M7 `event_buffer` 播种约定**：daemon `SessionRegistry.restore_factory`（server 注入 `_default_session_factory`，对「sqlite 已存在 id」走 `from_store`）+ `restore(session_id, session)`：把 `session.event_stream` 最近 `DEFAULT_BUFFER_SIZE`(200) 条事件播种进 `handle.event_buffer`，使后续 `attach` 的客户端 `_replay` 回放断点前上下文，与热切换回放一致。冷启动 `client --session <id>` / `--resume`（从 `SessionStore.list_sessions` 取最近 id 再 `SESSION_ATTACH`）均走此路径。
- **fork 恢复无特殊分支**：fork 子会话的 events 已含父前缀拷贝（M6.1 `SessionStore.fork` 的 INSERT…SELECT），`load`→`rebuild_messages` 与普通会话完全一致；`parent_session_id` 仅用于 `list_sessions` 展示血缘，恢复逻辑不区分。
- **测试**：`tests/test_session_recovery.py`（rebuild 映射 / 澄清映射 / 中断检测 / 中断丢弃悬空 / fork 前缀 / 跨重启 resume）+ `test_daemon.py::test_attach_restores_from_factory_and_seeds_buffer`。全量 `pytest` **392 passed**（M6.1 为 384，+8）。
- **踩坑**：`test_loop.py::test_basic_flow_records_events` 原先断言事件序列不含 user，M6.2 因新增 USER 事件更新其期望值（在序列首部加 `"user"`）。
